# Qoder CLI Native Dynamic Workflows — Exact Semantics, Limits, Headless Availability

**Method note:** The public doc (https://docs.qoder.com/en/cli/workflows.md, fetched 2026-07-04, 7.4KB) is an overview with no API reference. All exact semantics below were extracted by decompiling the shipped CLI: npm package `@qoder-ai/qodercli` **v1.0.37, published 2026-07-03** (tarball https://registry.npmjs.org/@qoder-ai/qodercli/-/qodercli-1.0.37.tgz, main bundle `package/bundle/qodercli.js`, 35.4MB Bun-bundled JS). Obfuscated strings use XOR key `QdPfjvqbh4vs` over base64 (decoder at bundle offset 26); I decoded the full embedded Workflow tool prompt, the vm bootstrap scripts, and the runtime. Byte offsets cited for verifiability. Release manifest confirms 1.0.37 latest (https://qoder-ide.oss-accelerate.aliyuncs.com/qodercli/channels/manifest.json).

**Headline: Qoder's Dynamic Workflows is a faithful, deep port of Claude Code's ultracode/dynamic-workflows feature** (compare https://code.claude.com/docs/en/workflows.md — "ultracode" is literally Claude Code's name for this feature: `/effort ultracode`, `ultracode` keyword trigger). Same script format, same helpers, same caps (16 concurrent / 1,000 lifetime), same determinism bans, same prefix-replay resume, same bundled `deep-research` workflow. It is NOT shallow.

## 1. Workflow tool surface (how the model invokes it)

Registered tool name: `"Workflow"` (constant `Ju="Workflow"` at offset 2097051; a `"RunWorkflow"` constant also exists — apparently unused/legacy). Registered generically in the tool registry when the feature gate is on (`this.isWorkflowsEnabled()&&A.registerTool(new U2A(this))`, offset 30243300) — **not TUI-specific**. `kind:"workflow"`, `isOutputMarkdown:true`, `maxResultSizeChars:100000`, tool result summary truncated at 8,000 chars ("[workflow result truncated]").

Tool input schema (offset ~29769000, decoded):
- `script` (string, `maxLength: 524288`): "Self-contained workflow script. Must begin with `export const meta = { name, description, phases }` as a pure literal followed by the script body using agent()/parallel()/pipeline()/phase()."
- `scriptPath` (string): path on disk; every invocation persists its script and returns scriptPath; "To iterate, edit that file with Write/Edit and re-invoke Workflow with the same scriptPath". Takes precedence over script and name. UNC paths (`\\\\`, `//`) rejected.
- `name` (string): "Name of a predefined workflow from the built-in, plugin, project, or user workflow registry." Name charset `[a-zA-Z0-9._:-]+`.
- `args` (any, verbatim): exposed as global `args`; JSON-looking strings are auto-parsed to objects/arrays (`aRe`); validated against `meta.inputSchema` if declared (error: "Workflow args do not match <name> meta.inputSchema").
- `resumeFromRunId` (string, must match `^wf_[a-z0-9-]{6,}$`): "Run ID of a prior same-session Workflow invocation to resume from. Completed agent() calls with unchanged prompt/options return cached results instantly; only edited or new calls re-run. Stop the prior run before resuming." Resuming a still-`running` run throws.
- `description`/`title`: explicitly "Ignored - set … in the script's `meta` block."

`validateParams` requires one of script/scriptPath/name/resumeFromRunId; pre-parses `script` with **acorn** (`ecmaVersion:"latest"`, `sourceType:"module"`, `allowAwaitOutsideFunction`, `allowReturnOutsideFunction` — top-level `return` is legal). `meta` must be a **pure literal** (no variables/calls/spreads/interpolation); required `name`+`description`; optional `title`, `whenToUse`, `phases` (`{title, detail, model?}` — "A phase entry may include model when that phase should use a specific model override"), `inputSchema` (JSON Schema for args).

`execute` is **fire-and-forget**: registers a background task `{id:"wf-<12hex>", type:"local_workflow", runId:"wf_<12hex>", …}`, writes manifest, returns immediately `llmContent`: "Workflow launched in the background." + taskId/runId/summary/transcriptDir/scriptPath (offset ~29774000). Completion is injected back into the session as a queued `task-notification` prompt (priority `"next"` if failed, `"later"` otherwise) containing Result/Failures/Usage (agents=, toolCalls=, durationMs=)/Artifacts (outputPath, scriptPath, transcriptDir, runId) and, on failed/stopped, Recovery instructions: "Run Workflow again with scriptPath=… and resumeFromRunId=…".

Permissions: per-workflow-name allow/ask/deny rules checked against the workflow's name (i.e., rules of the form `Workflow(name)`); otherwise `behavior:"passthrough"` with the message "Review dynamic workflow before running: …" → flows into the standard permission pipeline (LLM classifier input = scriptPath/name/2KB script preview). TUI shows the review dialog ("run it / view raw script / reject with feedback / cancel" per docs.qoder.com/en/cli/workflows.md).

## 2. Script-body API (from the embedded 6.3KB tool prompt, offset 29741620, + runtime at 29752000–29767000)

Sandbox-injected globals (fixed set, `L2n` at 29752559): `agent`, `parallel`, `pipeline`, `phase`, `log`, `workflow`, `args`, `budget`, `console` (log/info/debug mapped to log; warn/error prefixed `[warn]`/`[error]`), wrapped `setTimeout`/`clearTimeout` (cleared on abort). Nothing else — **plugins cannot add host functions**; plugins can only ship saved workflows (`<plugin>/workflows/` dir, source label `plugin:<name>`).

### agent()
`agent(prompt: string, opts?) | agent({prompt, ...opts})` → `Promise<any>`. Documented opts: `label?`, `phase?` (assign to a progress group; "use it inside pipeline()/parallel() stages to avoid races on the global phase() state"), `schema?` (JSON Schema object), `model?` (per-call model override), `isolation?: 'worktree'` ("runs the agent in a fresh git worktree… only for parallel file mutations that would otherwise conflict"), `agentType?` ("uses a custom subagent type from the same registry as the Agent tool"; aliases `type`/`subagent_type` accepted; default `general-purpose`; **remote agents rejected**: "is remote. Workflow currently supports local agents only."). Additional **undocumented runtime options** (normalizers `tHI`/`rHI`/`J8I`): `name`, `description`, `teamName`, `cwd` (working-dir override), `retries` (int, clamped 0–5 via `LXi=5`), `stallMs` (watchdog: no progress within stallMs → `WorkflowAgentStallError`, auto-retried up to 5 attempts), `skip: true` + `skipReason` (returns null, journaled `user-skip`). **No per-call `effort` or `temperature`** — those exist only on subagent *definitions* (`.qoder/agents/*.md` frontmatter supports `effort: low|medium|high|xhigh|max|int`, `model`, `temperature`, `maxTurns`, `timeoutMins`, `isolation: worktree` — https://docs.qoder.com/en/cli/subagent.md), so effort routing is achievable via `agentType`.

**Schema enforcement is real and strong** (`D2n` at 29731426): with `schema`, the subagent definition is cloned, a `StructuredOutput` tool is injected plus a system-prompt suffix ("You MUST call the StructuredOutput tool exactly once at the end. The tool input IS your answer."); errors if the subagent finishes without calling it or calls it ≠1 times; the result is validated against the schema (`QM.validate`) and `agent()` returns **the validated object** (not text). Without schema, returns final text. Returns `null` if the user skips the agent mid-run ("filter with .filter(Boolean)").

Failure semantics: an `agent()` that exhausts retries **throws** (recorded in `failures[]` as `agent[<n>] <label> failed: <msg>`); user can skip (→null) or retry (→`WorkflowAgentUserRetryError`, retried up to 5) individual agents live from the `/workflows` detail view via per-agent AbortControllers (abort reasons `user-skip` / `user-retry`).

### parallel() / pipeline()
- `parallel(thunks: Array<() => Promise<any>>)` — **barrier** via `Promise.allSettled`; a thunk that throws resolves to `null` in the result array and pushes `parallel[<i>] failed: <msg>` into failures+logs. Never fail-fast. TypeError if not an array of functions.
- `pipeline(items, stage1, stage2, …)` — **no barrier between stages**; all items processed concurrently (allSettled over items), each item flows through stages sequentially; each stage receives `(prevResult, originalItem, index)`; a stage returning `null` or throwing drops that item to `null` and skips its remaining stages (`pipeline[<i>] failed:` recorded). Tool prompt: "Default to pipeline() for multi-stage item-wise work. Use parallel() only when a stage genuinely needs all prior results together."
- Concurrency is enforced at the `agent()` level by a global semaphore: default `min(16, max(2, os.cpus().length - 2))` (`eHI`, offset 29765261); extra calls queue. Lifetime cap `maxAgents = 1000` per run (`AHI=1e3`) → error "…max agents (1000)". Both are internal `Y2n` options (host-side), **not script-settable**. Log cap 1,000 entries (`$8I`). vm sync-execution timeout 30s (`F2n=3e4`, applied to `script.runInContext(ctx,{timeout:30000})` — async awaits are unbounded).

### phase()/log()/args/budget/workflow()
- `phase(title)` — creates/reuses a named phase (index), later agents grouped under it; seeded from `meta.phases`. `log(message)` → `workflow_log` progress events.
- `budget: {total: number|null, spent(): number, remaining(): number}` — "the turn token target… pool shared by the main loop and workflows". **Caveat: in 1.0.37 the host (`lHI`) never passes a budget, so it is always the stub `{total:null, spent:0, remaining:Infinity}`** — API present, wiring incomplete. Agents throw "Workflow budget exceeded" if remaining ≤ 0.
- `workflow(nameOrRef | {name|scriptPath, args?}, args?)` — runs a **child workflow inline**, shares the parent's concurrency cap, agent counter, abort signal, and token budget; **nesting limited to one child level**; unknown names / unreadable scriptPath / child syntax errors throw (catchable). Child phases appear as kind `"child"`.
- Result of a run: `{result (top-level return value), logs, failures, agentCount, totalTokens, totalToolCalls, durationMs, error?}` → serialized to `output.json`.

## 3. Determinism restrictions and resume/caching

Sandbox bootstrap scripts (decoded from vm `runInContext` blobs at 29752994/29754366):
- `Date.now()` throws "Date.now() is unavailable in workflow scripts; pass timestamps via args."; no-arg `new Date()` and function-call `Date()` throw likewise; `new Date(<args>)`, `Date.parse`, `Date.UTC` allowed.
- `Math.random()` throws "Math.random() is unavailable in workflow scripts; pass deterministic values via args."
- `delete globalThis.WebAssembly; delete globalThis.ShadowRealm;` then `Object.freeze` on 28 intrinsics' prototypes + constructors + `Math`.
- Tool prompt states the reason verbatim: "unavailable because they would break **resume determinism**".

Resume/caching (`P2A` WorkflowJournal, offset ~29736000): append-only JSONL `journal.jsonl` per run. Cache key is a **hash chain**: `key_n = "v2:" + sha256(key_{n-1} + "\0" + prompt + "\0" + stableStringify({agentType, isolation, model, schema}))`. `nextCachedResult` replays sequentially; the first miss sets `prefixMissed` and disables all later hits → exactly "the longest unchanged prefix of agent() calls returns cached results; the first edited or new call and everything after it runs live. Same script plus same args should be a full cache hit." Journal records `started`/`event`(attempt_started/retry/skipped/error)/`result` entries. On resume, the old run's journal is the read path and the new run gets a fresh journal. Persistence layout: `<project>/.qoder/sessions/<sessionId>/workflows/runs/<wf_runId>/{manifest.json, journal.jsonl, output.json}` + transcripts dir; scripts at `<project>/.qoder/sessions/<sessionId>/workflows/scripts/<slug>-<runId>.js` (offsets 28105876–28106100). Tool description says resume is "same-session" (manifests are addressed by sessionId), but artifacts are plain files on disk — resuming after process restart works if you resume the same session (`qodercli -r <sessionId>`).

## 4. Headless (`qodercli -p`) and SDK availability

- The Workflow tool is registered in the **shared** tool registry (gate-only condition), so it exists in print mode and SDK sessions, not just TUI. `/workflows` (alt `workflow-tasks`, "Show workflow tasks panel") and `/tasks` are TUI-only monitoring UIs (https://docs.qoder.com/en/cli/using-cli.md documents `-p`, `--output-format text|json|stream-json`, `--yolo`, `--max-turns`, `--allowed-tools`).
- Feature gate: `isWorkflowsEnabled() → sDn()` = a **remote config-service flag** keyed `"workflows"` fetched after auth ("[config-service] workflows feature gate value received"), local initial default `true` (`Yki=!0`), env kill-switch `QODER_FEATURE_WORKFLOWS_DISABLE` (offset 26693308). A separate env fallback `Ta("WORKFLOWS")` reads `<QODER_>FEATURE_WORKFLOWS` (default **false**) and is used only where no config object exists (slash-command list building, offset 30903398). Tool refuses with "Workflow feature gate is disabled." So headless/SDK availability depends on the account-level remote gate; the docs page being live (published in llms.txt) suggests broad rollout as of July 2026.
- Approval in headless: no name-matching allow rule → `passthrough` → standard permission pipeline; with `--yolo`/`bypassPermissions` it runs immediately; with `dontAsk` it is denied rather than prompted. Child agents' tool calls go through normal permissions/hooks; in workflows, an "ask" on the *agent dispatch itself* throws "Agent invocation requires interactive approval" (D2n) — so headless runs should pre-authorize (allowlist or yolo).
- SDK/stream-json monitoring: SDK message stream includes `SDKTaskStartedMessage {subtype:'task_started', task_id, task_type?, workflow_name?, prompt?}`, `SDKTaskProgressMessage {usage:{total_tokens, tool_uses, duration_ms}, last_tool_name?, summary?}`, `SDKTaskNotificationMessage {status:'completed'|'failed'|'stopped', output_file, summary, usage?}` (https://docs.qoder.com/en/cli/sdk/references.md); the same zod schemas exist in the bundle (offset 27378500) for `--output-format stream-json`. Workflow child agents run with `emitSdkSideEvents:true, emitSdkTaskNotification:false, registerTask:false`. SDK also exposes `TaskOutput`/`TaskStop` tools to poll/stop background tasks. There is **no dedicated SDK "runWorkflow" API** — you invoke via prompt (the model calls the Workflow tool) and monitor via task events + on-disk `output.json`/`manifest.json`.
- Unverified: whether a bare `qodercli -p "…"` process stays alive until the background workflow finishes. Mechanism (completion enqueues a `task-notification` prompt into the session queue) suggests the turn continues to a final summarization, matching Claude Code's documented `-p` support, but I found no explicit drain/wait code.

## 5. Sandbox implementation

**Node's `node:vm`** (`import MNe from"node:vm"`, offset 29752330) in-process — not QuickJS, not a worker, not isolated-vm. Script wrapped as `(async () => {\n<body>\n})()`, compiled as `new vm.Script(..., {filename:"workflow.js"})`, context from `vm.createContext({...hostFns, ...wrappedTimers})`, hardened by the two bootstrap scripts (intrinsic freeze + Date/Math.random bans + WebAssembly/ShadowRealm removal). No fs/shell/network/require/process/MCP — confirmed both in docs and code. Note `node:vm` is not a hard security boundary; Qoder relies on it plus review-before-run. Host-function extension points: none for plugins (fixed global set); plugins contribute only saved workflow files. Registry precedence: project `.qoder/workflows` > user `~/.qoder/workflows` > plugin > built-in (docs), with one built-in: **`deep-research`** (full 5-phase Scope/Search/Fetch/Verify/Synthesize script with JSON schemas embedded, offset ~28107300 — near-identical in design to Claude Code's `/deep-research`).

## Recommendation: INTEGRATE (thin compatibility layer), do not replace

Qoder's native runtime already implements the hard parts of ultracode — schema-enforced agent results, semaphore concurrency, deterministic sandbox, hash-chained prefix-replay resume, nested workflows, per-agent live control — with the *same script dialect*. A portable ultracode design targeting Qoder should:
1. **Emit/install scripts into `.qoder/workflows/*.js`** using the shared dialect (meta + agent/parallel/pipeline/phase/log/args). The dialect is source-compatible with Claude Code's `.claude/workflows/*.js` except: no `budget` reliance (stubbed in Qoder), no Claude-specific agentTypes, and per-call `effort` must be encoded as a custom `agentType` definition in `.qoder/agents/*.md` (frontmatter `effort:`).
2. **Trigger headlessly** via `qodercli -p "Run the <name> workflow with args …" --output-format stream-json` plus `--yolo` or a `Workflow(<name>)` allow rule (and pre-allowlisted child-agent tools, since "ask" throws in workflows); parse `task_started/task_progress/task_notification` events (`workflow_name`, `output_file`) or poll `<project>/.qoder/sessions/<sid>/workflows/runs/<wf_*>/output.json` + `manifest.json` (status field) for machine monitoring.
3. **Namespace defensively**: don't define workflows named `deep-research`; don't create a tool/command named `Workflow`, `workflows`, or `workflow-tasks`; keep any external-orchestrator fallback's on-disk state out of `.qoder/sessions/*/workflows/`.
4. **Fallback path** (only needed if the remote `"workflows"` gate is off for a target account): keep the external orchestrator as a degraded mode driving `qodercli -p` + `--agents` JSON injection per subagent; detect gate state by probing for the tool ("Workflow feature gate is disabled." result) at setup time.
5. Gaps worth papering over in the compat layer: no auto-orchestration effort mode (implement keyword→prompt rewriting client-side), stubbed `budget` (inject budget via args), same-session resume addressing (persist sessionId alongside runId so resume uses `-r <sessionId>` + `resumeFromRunId`).

## UNCERTAINTIES
- Whether a bare `qodercli -p` process waits for background workflow completion before exiting: the completion path enqueues a 'task-notification' prompt into the session queue (suggesting the turn continues until the result arrives, matching Claude Code's documented -p support), but I found no explicit wait/drain code and could not run the CLI (not installed; read-only phase).
- Whether opts.phase on agent() is actually honored by the 1.0.37 runtime: the embedded tool prompt documents it, but I could not locate the E.phase handling in the minified executor (events use the current global phase index). Possibly handled elsewhere or a doc-ahead-of-code gap.
