# Cross-Agent Standards & Prior Art for a Portable Multi-Agent Orchestration Layer (as of 2026-07)

## 1. MCP (Model Context Protocol) — state as of mid-2026

### Version landscape
- **Current stable spec: `2025-11-25`** (published at the protocol's first anniversary). Key additions vs 2025-06-18 (changelog: https://modelcontextprotocol.io/specification/2025-11-25/changelog):
  - **Experimental Tasks** (SEP-1686) — durable async request wrapper (details below).
  - **Tool calling in sampling** via `tools` and `toolChoice` params (SEP-1577).
  - **URL-mode elicitation** (SEP-1036) + richer `ElicitResult`/`EnumSchema` incl. multi-select enums (SEP-1330) + defaults in elicitation schemas (SEP-1034).
  - JSON Schema 2020-12 as default dialect (SEP-1613); icons metadata (SEP-973); OIDC discovery, incremental scope consent (SEP-835), OAuth Client ID Metadata Documents (SEP-991).
  - Input-validation errors returned as **Tool Execution Errors** (isError result) not protocol errors, to enable model self-correction (SEP-1303).
  - **Polling SSE streams**: servers may disconnect SSE at will; resumption always via GET + event IDs (SEP-1699, clarified in #1847).
  - stdio servers may use stderr for all logging (PR #670).
- **Next version: 2026-07-28 release candidate** (RC locked May 21, 2026; final publication July 28, 2026 — i.e., ~3 weeks after "now"). Source: https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/ . Headline changes:
  - **Stateless protocol core**: `initialize`/`initialized` handshake and `Mcp-Session-Id` header REMOVED (SEP-2567, SEP-2575); client metadata travels in `_meta` on every request; new `server/discover` method for on-demand capability discovery.
  - **Streamable HTTP**: required `Mcp-Method` and `Mcp-Name` headers for LB routing (SEP-2243); `ttlMs`/`cacheScope` on list/resource responses (SEP-2549); W3C Trace Context in `_meta` with keys `traceparent`, `tracestate`, `baggage` (SEP-414).
  - **Tasks graduate from experimental core to an official extension** (SEP-2663), redesigned to work statelessly: server answers `tools/call` with a task handle; client drives via `tasks/get`, `tasks/update`, `tasks/cancel`; **task creation is server-directed** (server decides when a call runs as a task); **`tasks/list` removed**.
  - **Elicitation redesigned for statelessness** (SEP-2260, SEP-2322): servers return `InputRequiredResult` with `inputRequests` + `requestState`; client re-issues the call with `inputResponses` + echoed state (no held-open SSE stream).
  - **Extensions framework first-class** (SEP-2133): reverse-DNS IDs, capability-map negotiation, `ext-*` repos, independent versioning. Two official extensions: **Tasks** and **MCP Apps** (sandboxed-iframe HTML UIs, SEP-1865).
  - **Deprecations (12-month windows, SEP-2577): Roots, Sampling, and Logging are deprecated** — sampling's replacement is "direct LLM provider API integration"; logging's replacement is stderr (stdio) / OpenTelemetry. **Do not architect a new orchestration layer around MCP sampling.**
  - Tool input schemas may now use `oneOf/anyOf/allOf`, conditionals, `$ref` (still `type:"object"` root) (SEP-2106); `structuredContent` accepts any JSON value; missing-resource error becomes standard `-32602` (SEP-2164).
- 2026 roadmap (https://blog.modelcontextprotocol.io/posts/2026-mcp-roadmap/): priorities = transport scalability (stateless sessions, `.well-known` discovery; no new transports), Tasks iteration (retry semantics, result-expiry policies), governance (contributor ladder, WG delegation), enterprise readiness (as extensions). "On the horizon": triggers/event-driven updates, streamed result types.

### Tasks (2025-11-25, experimental) — the machinery for long-running work
Spec: https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks
- **Model**: requestor (client or server) augments any supported request with `"task": {"ttl": <ms>}` in params. Receiver immediately returns a `CreateTaskResult`: `{task:{taskId, status:"working", statusMessage?, createdAt, lastUpdatedAt, ttl, pollInterval}}` — the real result comes later via `tasks/result`.
- **Methods**: `tasks/get` (poll status), `tasks/result` (**blocks** until terminal; returns exactly what the underlying request would have returned, e.g. `CallToolResult`), `tasks/list` (cursor-paginated), `tasks/cancel` (must reject if already terminal with `-32602`).
- **Statuses**: `working` → (`input_required` ↔ `working`) → terminal `completed` | `failed` | `cancelled`. Tool results with `isError:true` map to `failed`. `statusMessage` carries human-readable/diagnostic text in any state.
- **Capabilities**: server declares `capabilities.tasks = {list:{}, cancel:{}, requests:{tools:{call:{}}}}`; client-side mirror for `sampling.createMessage` and `elicitation.create`. Per-tool negotiation via `execution.taskSupport: "required" | "optional" | "forbidden"` in `tools/list` (default forbidden; wrong mode ⇒ `-32601`).
- **`_meta` conventions**: all task-related messages MUST carry `_meta["io.modelcontextprotocol/related-task"] = {taskId}` ; servers MAY put `_meta["io.modelcontextprotocol/model-immediate-response"]` (a string) in `CreateTaskResult` so the host can hand the model an immediate tool result while the task runs.
- **Lifecycle rules**: receiver generates unique string taskIds (cryptographically random when unauthenticated); MUST include `createdAt`/`lastUpdatedAt` (ISO 8601); MAY override requested `ttl` (actual ttl or `null` returned); after ttl elapses task+results may be deleted; `pollInterval` (ms) is a polling hint clients SHOULD respect; optional `notifications/tasks/status` push (full Task object) which clients MUST NOT rely on.
- **input_required flow**: server parks task in `input_required`; client SHOULD preemptively call `tasks/result`, which opens a channel over which the server sends `elicitation/create` (tagged with related-task); after input, task returns to `working`.
- **Errors**: unknown/expired taskId ⇒ `-32602`; internal ⇒ `-32603`; server MAY require task augmentation (`-32600` "Task augmentation required for tools/call requests").
- **Security**: bind tasks to auth context where available; rate-limit task ops; don't declare `tasks.list` if requestors are unidentifiable; enforce concurrent-task and max-ttl limits.

### Progress & cancellation (stable utilities)
- **Progress** (https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/progress): requester puts `_meta.progressToken` (string|int, unique across active requests) in the request; receiver sends `notifications/progress` with `{progressToken, progress (MUST monotonically increase, may be float), total?, message?}`. For task-augmented requests, **the original progressToken stays valid for the whole task lifetime** and notifications MUST stop at terminal status. Rate-limit; may omit total.
- **Cancellation** (https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/cancellation): `notifications/cancelled` `{requestId, reason?}`; receiver SHOULD stop work and not respond; race conditions must be tolerated; `initialize` not cancellable; **for tasks use `tasks/cancel` instead**.

### Transports
- **stdio**: newline-delimited JSON-RPC over stdin/stdout; server logging via stderr; simplest for a local orchestrator embedded in a coding agent; notifications (progress) work fine. **Streamable HTTP**: POST per message, optional SSE streams for server→client messages, resumability via SSE event IDs + `Last-Event-ID`, polling-style SSE allowed (SEP-1699); 403 on bad Origin. Spec: https://modelcontextprotocol.io/specification/2025-11-25/basic/transports . The 2026-07-28 RC makes HTTP stateless (no session header) — design servers to not depend on session affinity.

### Recommended shape for a long-running "workflow" tool over MCP (synthesis)
1. **Dual-mode**: expose the workflow tool with `execution.taskSupport:"optional"` and declare `tasks` capability; for clients without task support (still the majority in mid-2026 — flag: client support matrix unverified), ALSO expose an app-level trio: `workflow_start` (returns `{workflowId}` fast), `workflow_status`, `workflow_result` — this is the same pattern Tasks formalizes and maps 1:1 onto the Tasks extension later.
2. **Return the task handle immediately** with `pollInterval` tuned to your step cadence, `ttl` long enough to survive client restarts, and `model-immediate-response` text like "Workflow started; 5 agents running; poll for status."
3. **Progress**: honor `progressToken` for the entire task; use `message` for step-level human-readable state ("agent 3/5 done: tests green"); use monotonically increasing `progress` with `total` = number of workflow steps.
4. **Partial results**: MCP has no first-class partial-result channel — use (a) `statusMessage` on `tasks/get`, (b) progress `message`, and (c) write a journal file / expose it as an MCP **resource** and return `resource_link` content blocks in the final `CallToolResult`; final result should include `structuredContent` conforming to the tool's `outputSchema`.
5. **Human-in-the-loop**: park in `input_required` + elicitation (2025-11-25) but be ready for the 2026-07-28 `InputRequiredResult`/`inputResponses` re-issue pattern. **Avoid sampling** (deprecated in RC) — call model providers directly for internal LLM steps.
6. **Cancellation**: implement `tasks/cancel` as best-effort kill of child agent processes; keep task record in `cancelled` state; treat `notifications/cancelled` for non-task calls as SIGTERM to the workflow.
7. **Timeouts**: never rely on a single long `tools/call` — hosts enforce tool timeouts (e.g. Claude Code's MCP tool timeout / `MCP_TIMEOUT`); anything over ~1 min should be a task/app-level job.

## 2. Agent Skills standard

- **Site/spec**: https://agentskills.io (spec at https://agentskills.io/specification); repo https://github.com/agentskills/agentskills (reference validator: `skills-ref validate ./my-skill`). Originally developed by Anthropic, **released as an open standard Dec 18, 2025** (https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills, https://simonwillison.net/2025/Dec/19/agent-skills/).
- **Format**: a skill = a directory containing `SKILL.md` (YAML frontmatter + Markdown body) plus optional `scripts/`, `references/`, `assets/`.
- **Frontmatter fields (exact, from spec)**:
  - `name` (required): 1–64 chars, lowercase alphanumerics + hyphens, no leading/trailing/consecutive hyphens, **must match parent directory name**.
  - `description` (required): 1–1024 chars; what it does + when to use it.
  - `license` (optional): short license name or bundled-file reference.
  - `compatibility` (optional): 1–500 chars; environment requirements (product, packages, network).
  - `metadata` (optional): arbitrary string→string map (put product-specific keys here).
  - `allowed-tools` (optional, **Experimental**): space-separated pre-approved tools, e.g. `allowed-tools: Bash(git:*) Bash(jq:*) Read`; support varies by implementation.
- **Progressive disclosure (3 levels)**: (1) metadata only (~100 tokens: name+description) loaded at startup for all skills; (2) full `SKILL.md` body (<5000 tokens recommended, keep under 500 lines) loaded on activation; (3) bundled files loaded on demand. File references: relative paths from skill root, keep one level deep.
- **Adoption (mid-2026, per the client carousel on agentskills.io/home — 40+ clients, each with an instructions URL)**: Claude Code (https://code.claude.com/docs/en/skills), Claude apps, **OpenAI Codex** (https://developers.openai.com/codex/skills/), **Gemini CLI** (https://geminicli.com/docs/cli/skills/), **Cursor** (https://cursor.com/docs/context/skills), **opencode** (https://opencode.ai/docs/skills/), **GitHub Copilot** (https://docs.github.com/en/copilot/concepts/agents/about-agent-skills), **VS Code**, Amp, Goose (Block), OpenHands, JetBrains Junie, Roo Code, Factory, Kiro, TRAE (ByteDance), Qodo, Letta, Mux (coder/mux), Emdash, Superconductor, Ona, Firebender, Databricks Genie Code, Snowflake Cortex Code, Spring AI, Mistral Vibe, Tabnine, fast-agent, nanobot (HKUDS), Laravel Boost, VT Code, pi, Workshop, Deep Code, and more. Anthropic's example-skill repo: https://github.com/anthropics/skills.
- Implication for a portable orchestrator: ship your orchestration layer's knowledge/prompts as skills (SKILL.md folders) and they load in every major coding agent; keep product-specific config under `metadata`.

## 3. AGENTS.md convention

- **Site**: https://agents.md — "a README for agents"; plain Markdown, **no required fields**; stewarded by the **Agentic AI Foundation under the Linux Foundation** (born from OpenAI Codex, Amp, Google Jules, Cursor, Factory collaboration); **~60,000+ open-source repos** use it; repo: https://github.com/agentsmd/agents.md.
- **Precedence**: nested AGENTS.md supported for monorepos — **closest file to the edited file wins**; explicit user chat prompts override file instructions.
- **Who reads it (30+ tools per agents.md)**: OpenAI Codex (https://developers.openai.com/codex/guides/agents-md), Google Jules, **Gemini CLI**, Cursor, Aider, VS Code / GitHub Copilot, Devin, Windsurf, Amp, Roo Code, JetBrains Junie, UiPath Autopilot, Zed, Warp, Factory, Semgrep, Ona, etc.
- **Qoder**: yes — Qoder rules are AGENTS.md-compatible; auto-recognized from project dir with no config; on conflict, Qoder's own `.qoder/rules` content takes precedence (https://docs.qoder.com/user-guide/rules).
- **Claude Code**: primary file is CLAUDE.md; several 2026 sources (e.g. https://pnote.eu/notes/agents-md/) claim Claude Code now also reads AGENTS.md natively, but other guides still describe CLAUDE.md-only + symlink workaround — see uncertainties.

## 4. Prior art — OSS orchestrators of coding-agent CLIs

**Headless CLI substrates every orchestrator builds on** (the "structured results" channel):
- **Claude Code**: `claude -p "<prompt>" --output-format json|stream-json` (stream-json = NDJSON events, requires `--verbose`; `--include-partial-messages` for token deltas); every event carries `session_id`; resume via `--resume <session-id>` / `--continue`; final `result` event includes `total_cost_usd`. Docs: https://code.claude.com/docs/en/headless. Also the **Claude Agent SDK** (`@anthropic-ai/claude-agent-sdk`, Python equivalent): `query()` returns an async generator of typed messages; programmatic **subagents** with per-agent tools/models (https://platform.claude.com/docs/en/agent-sdk/subagents); V2 preview replaces generators with `send()`/`stream()` per turn (https://platform.claude.com/docs/en/agent-sdk/typescript-v2-preview). SDK reference: https://code.claude.com/docs/en/agent-sdk/typescript.
- **Codex CLI**: `codex exec --json` emits JSONL events `thread.started`, `turn.started`, `turn.completed`, `turn.failed`, `item.*` (agent messages, reasoning, command executions, file changes, MCP tool calls, web searches, plan updates), `error`; **`--output-schema <schema.json>`** forces the final message to conform to a JSON Schema; resume via `codex exec resume --last` or `codex exec resume <SESSION_ID>`. Docs: https://developers.openai.com/codex/noninteractive.
- **Gemini CLI**: extensions system for packaging (below); non-interactive `-p` mode exists (JSON output flag unverified — see uncertainties).

**Projects**:
- **claude-flow → renamed "ruflo"** (https://github.com/ruvnet/ruflo; legacy https://github.com/ruvnet/claude-flow): MIT. v3.16.3 (July 2026) — a security release fixing an **MCP-bridge RCE**. Positions as "agent meta-harness": MCP server exposing ~210 tools + 27-hook system + swarm topologies (hierarchical/queen-led, mesh, adaptive; Raft/Byzantine consensus claims) wrapped around Claude Code (and now Codex etc.). Spawns each agent (Claude Code default; Codex, Gemini, Aider, any command) in a **tmux session + git worktree**; pause (`c` checkout) / resume (`r`); diff tab + commit/push. **No structured results** — output is git branches/diffs; experimental `autoyes` auto-accept.

"Large numbers of agents in parallel": auto git-worktree per agent + tmux session + auto dev-server with port ranges from `uzi.yaml` (`devCommand`, `portRange`); real-time status watching; `uzi checkpoint` merges a winner back to main. Single-agent-looking front ("Mayor") that spawns and supervises specialized worker agents (Polecats, Refinery, Witness, Deacon, Dogs; human "Overseer") — primarily Claude Code instances. **Persistence/journaling via "Beads"** (gastownhall/beads, MIT): a git+SQLite-backed issue tracker where a "bead" (id, description, status, assignee) is the atomic unit of work — work state survives agent restarts; git-backed hooks re-hydrate agents. - **"gnattyclaw"**: no such project found (see uncertainties). Adjacent finds: **Gnap** (git-native agent protocol: shared git repo as todo/doing/done task board, no orchestrator process), **claw-empire** (https://github.com/GreenSheep01201/claw-empire, multi-CLI "virtual company"), **HKUDS/ClawTeam** (swarm-of-CLIs), **AgentWrapper/agent-orchestrator** (plans tasks, spawns agents, auto-handles CI/merge conflicts), **coder/mux** (parallel agents, isolated workspaces, browser/desktop), **Emdash** (worktree-per-agent desktop app). Curated list: https://github.com/andyrewlee/awesome-agent-orchestrators.

**Pattern summary**: two families — (A) *workspace multiplexers* (claude-squad, uzi, vibe-kanban, container-use, mux, Emdash): subprocess CLIs + git worktree/container isolation, results as branches/diffs, little structured data; (B) *programmatic orchestrators* (mcp-agent, OpenAI Agents SDK, Claude Agent SDK, claude-flow/ruflo, Gas Town): SDK or MCP-tool spawning, structured results (JSON events/output schemas), explicit memory/journals (SQLite, Temporal, beads). A portable layer should combine A's process/CLI isolation (spawn `claude -p --output-format stream-json`, `codex exec --json --output-schema`) with B's journaling and typed results.

## 5. Sandboxed JS execution for orchestration scripts in a Node.js tool

Reference comparison: Simon Willison's LLM-generated "JavaScript Sandboxing Research" (Mar 22, 2026, https://simonwillison.net/2026/Mar/22/javascript-sandboxing-research/ + https://github.com/simonw/research `javascript-sandboxing-research`): conclusion — built-in Node tools (vm, resourceLimits) are insufficient (prototype escapes, weak memory enforcement); **isolated-vm and quickjs-emscripten are "best-in-class in-process sandboxing"**; for adversarial code combine hardened engines + worker threads + Node Permission Model + OS sandboxing (defense in depth).

- **node:vm**: NOT a security mechanism per Node's own docs (https://nodejs.org/api/vm.html); trivially escapable via constructor/prototype chains; `timeout` only guards synchronous execution (microtasks/promises escape it); no memory limits. Fine only as a *scoping* device for trusted scripts. (Also: https://dev.to/dendrite_soup/nodevm-is-not-a-sandbox-stop-using-it-like-one-2f74.)
- **vm2**: discontinued 2023, resurrected Oct 2025, then **another critical sandbox-escape RCE in Jan 2026 (CVE-2026-22709)** (https://www.endorlabs.com/learn/cve-2026-22709-critical-sandbox-escape-in-vm2-enables-arbitrary-code-execution, https://semgrep.dev/blog/2026/calling-back-to-vm2-and-escaping-sandbox/). Do not use.
- **isolated-vm** (https://github.com/laverdet/isolated-vm): true V8 isolates — separate heaps, no shared prototype chain, real memory limits + CPU timeouts. Costs: native addon (node-gyp, Node-version coupling), **maintenance-mode** posture (README warns to only use with a dedicated security understanding; author has limited bandwidth), clunky value marshaling (`ExternalCopy`/`Reference`), async host-function bridging is manual. Fast (native V8) — right choice when script CPU perf matters and you can own the native-dep burden.
- **quickjs-emscripten** (https://github.com/justjake/quickjs-emscripten): MIT; QuickJS (or **quickjs-ng**, better ES2023) compiled to WASM — the sandbox boundary is the WASM heap, immune to V8-specific escapes. Best fit for an orchestration DSL:
  - **Host function injection**: `context.newFunction(name, fn)` → set on globals; exactly how you'd expose `agent()`, `parallel()`, `readResult()`.
  - **Async**: ASYNC build variants (Asyncify) let guest code `await` host promises (`newAsyncifiedFunction`); limitation: an asyncified module can suspend for **one async call at a time** — so implement `parallel()` on the host side (host fans out, guest awaits one aggregate promise). `runtime.executePendingJobs()` gives the host explicit, deterministic control of the microtask queue.
  - **Resource limits**: `runtime.setInterruptHandler()` (+ `shouldInterruptAfterDeadline()`), `setMemoryLimit()`, `setMaxStackSize()`.
  - **Determinism**: you construct the global environment; don't expose host `Date`/`Math.random` — override guest globals with seeded/frozen versions (QuickJS's own Date.now/Math.random exist but can be replaced at context setup). Combined with host-controlled job scheduling this enables deterministic replay of orchestration scripts.
  - **Serialization**: values cross as handles; `context.dump(handle)` / `unwrapResult`; JSON-roundtrip marshaling is the practical pattern; handles need explicit `.dispose()` (or `using`/`Scope`).
  - Status: active, 0.x ("occasional breaking API changes").
- **SES / Hardened JS** (`ses` from Endo: https://github.com/endojs/endo, https://hardenedjs.org/): `lockdown()` freezes all intrinsics (kills prototype pollution), `new Compartment(endowments)` gives per-script globals; `harden()` for shared objects. **Compartments get powerless shared intrinsics — `Date.now()`/`Math.random()` are denied unless you explicitly endow them** (https://docs.agoric.com/guides/js-programming/hardened-js) — the cleanest determinism story, and injecting `agent()`/`parallel()` is trivial (plain endowments, full native async/await, no serialization boundary). Production users: MetaMask (snaps), Agoric; SES 2.0.0 fixed a NaN-bit-pattern side channel. **Weaknesses**: same process/heap ⇒ no memory/CPU metering (an infinite loop blocks your event loop — pair with a worker thread + watchdog), and engine (V8) bugs are in-scope. Good when scripts are "untrusted-ish" (user-authored, not adversarial).
- **Deno as subprocess** (https://docs.deno.com/runtime/fundamentals/security/): OS-process isolation + deny-by-default permissions (`--allow-read=...`, `--allow-net=...`, etc.). Caveats: `--allow-run` subprocesses escape the sandbox entirely (`--allow-run=deno` ⇒ trivial full escape); host functions can't be injected directly — you must build an RPC bridge (stdio JSON-RPC / HTTP) for `agent()`/`parallel()`, which adds latency and serialization but gives crash isolation and lets scripts use a real stdlib. Determinism not built in. Deno also shipped a hosted "Deno Sandbox" product with Deno Deploy GA (Feb 2026).
- **ShadowRealm**: compared in the simonw research; not a security boundary (same process, shared platform bugs, no resource limits) and still not broadly available unflagged — not a candidate.

**Fit-for-purpose ranking for an `agent()`/`parallel()` orchestration DSL**: (1) **quickjs-emscripten (quickjs-ng, ASYNC variant)** — WASM boundary + interrupt/memory limits + host-controlled scheduling + easy banning of Date/Math.random ⇒ determinism & replay; (2) **SES in a worker thread** if you want zero-marshaling ergonomics and full async, accepting no hard resource limits; (3) **isolated-vm** if raw V8 speed is required; (4) **Deno subprocess** if scripts need npm/stdlib and you accept an RPC bridge. Avoid node:vm-as-sandbox and vm2 outright.


## KEY FACTS
