# GAP: MCP long-running tool behavior in Codex and Qoder as MCP hosts: timeout ceilings, progress-notification handling, and Tasks-extension support

# MCP long-running tool behavior: Codex, Qoder CLI, Gemini CLI (verified from source, 2026-07-03/04)

## 1. Codex (openai/codex, codex-rs — main branch @ 2026-07-04; latest stable rust-v0.142.5, 2026-07-01)

Codex's MCP client is now split across `codex-rs/codex-mcp` (connection manager, crate `codex_mcp::McpConnectionManager`) and `codex-rs/rmcp-client` (crate `codex-rmcp-client`, built on the official Rust SDK `rmcp = "1.8.0"` per `codex-rs/Cargo.toml` line 376). The old `core/src/mcp_connection_manager.rs` is gone.

### (d) tool_timeout_sec: default, type, ceiling
- Config key: `[mcp_servers.<name>] tool_timeout_sec` in `~/.codex/config.toml`. Type is `Option<Duration>` deserialized from an **f64 number of seconds** via `Duration::try_from_secs_f64` (`codex-rs/config/src/mcp_types.rs`, `mod option_duration_secs`, lines 464-488; field at lines 192-193 and 277-279). https://github.com/openai/codex/blob/main/codex-rs/config/src/mcp_types.rs
- **Very large values are accepted**: any non-negative finite f64 up to u64::MAX seconds (~5.8e11 years) parses; NaN/negative/infinite fail config parse. `tool_timeout_sec = 86400` (24 h) or larger works. **There is no "disable timeout" value**: omitting the key applies the default (`.unwrap_or(DEFAULT_TOOL_TIMEOUT)`, `codex-rs/codex-mcp/src/rmcp_client.rs` lines 332-335); `0` gives a zero-duration timeout (instant failure), not infinity.
- **Default is now 300 s, not 60 s**: `pub(crate) const DEFAULT_TOOL_TIMEOUT: Duration = Duration::from_secs(300);` and `DEFAULT_STARTUP_TIMEOUT = 30s` (`codex-rs/codex-mcp/src/rmcp_client.rs` lines 85-86). History verified via tags: rust-v0.98.0 = 60 s (in old `core/src/mcp_connection_manager.rs`), rust-v0.110.0 through rust-v0.135.0 = 120 s, rust-v0.142.5 and main = 300 s. **Official docs at https://developers.openai.com/codex/mcp are stale** (still say tool_timeout_sec default 60, startup_timeout_sec default 10). `startup_timeout_sec` (f64) and `startup_timeout_ms` (u64) both exist for startup; there is no `tool_timeout_ms`.
- The same `tool_timeout` is applied to tools/call, tools/list, resources/list, resources/read (`codex-rs/codex-mcp/src/connection_manager.rs` lines 548, 606, 678, 751, 789-826).

### (a) Do progress notifications reset/extend the per-call timeout? **NO.**
- The timeout is Codex's own wrapper `active_time_timeout(duration, pause_state, operation)` (`codex-rs/rmcp-client/src/rmcp_client.rs` lines 181-224) applied in `run_service_operation_once` (lines ~1036-1056). The only thing that pauses the clock is a **pending elicitation request** (`ElicitationPauseState` — while the server has an elicitation waiting on the user, elapsed time doesn't count; remaining time resumes after). Test name: `active_time_timeout_pauses_while_elicitation_is_pending`.
- `notifications/progress` is handled by `LoggingClientHandler::on_progress`, which **only writes a tracing `info!` log line** (token, progress, total, message) — no timer interaction, no UI event (`codex-rs/rmcp-client/src/logging_client_handler.rs`).
- Irony: rmcp 1.8.0 natively supports `PeerRequestOptions { timeout, reset_timeout_on_progress, max_total_timeout }` and auto-injects `_meta.progressToken` into **every** outgoing request (`rust-sdk/crates/rmcp/src/service.rs`, `send_request_with_option`, lines ~618-660 — `request.get_meta_mut().set_progress_token(...)`). Codex uses `PeerRequestOptions::no_options()` setting only `.meta` (`codex-rs/rmcp-client/src/rmcp_client.rs` `call_tool`, lines ~599-656), so rmcp's timeout/progress-reset machinery is completely bypassed. Net: **servers do receive a progressToken and can send correlated progress, but it never extends Codex's deadline.**

### (b) Behavior on timeout
- `active_time_timeout` expiry → `ClientOperationError::Timeout` → error string `"timed out awaiting tools/call after {duration:?}"` (e.g. `after 300s`), wrapped with anyhow context `"tool call failed for `{server}/{tool}`"` (`codex-rs/codex-mcp/src/connection_manager.rs` line ~753), then `core/src/mcp_tool_call.rs::execute_mcp_tool_call` does `.map_err(|e| format!("tool call error: {e:?}"))`.
- The final `CallToolResult::from_error_text` (`codex-rs/protocol/src/models.rs` lines 1989-2006) returns to the model: `content: [{"type":"text","text":"tool call error: tool call failed for `srv/tool`\n\nCaused by:\n    timed out awaiting tools/call after 300s"}], is_error: true`. **The turn continues; the model sees the error and can react.**
- **No `notifications/cancelled` is sent.** On timeout the operation future (holding rmcp's `RequestHandle` inside `await_response`) is simply dropped. rmcp only sends `notifications/cancelled` from its own timeout paths (`send_timeout_cancel_notification`) or an explicit `RequestHandle::cancel()` — neither is reached; there is no Drop impl that cancels. The MCP server is never told; a stdio server process keeps running the tool and its eventual response is discarded (orphaned work).
- Transport retries (`run_service_operation_with_transient_retries`) apply **only to `tools/list`** (`is_retryable_tools_list_error`), never tools/call; streamable-HTTP session-expiry (404) recovery re-initializes and retries the operation once.

### (c) Tasks (MCP 2025-11-25) support: **NONE.**
- Codex explicitly pins the initialize handshake to `ProtocolVersion::V_2025_06_18` (`codex-rs/codex-mcp/src/rmcp_client.rs` line ~888, `InitializeRequestParams::...with_protocol_version(ProtocolVersion::V_2025_06_18)`), even though rmcp 1.8's `ProtocolVersion::LATEST = V_2025_11_25` (`rust-sdk/crates/rmcp/src/model.rs` lines 163-174). No `tasks/get`, `tasks/result`, `tasks/cancel`, or task-augmentation code exists anywhere in codex-rs (the `cloud-tasks*` crates and `core/src/tasks` are Codex Cloud / internal agent tasks, unrelated to MCP Tasks).

## 2. Qoder CLI (npm `@qoder-ai/qodercli` v1.0.37, published 2026-07-03; bundle inspected directly)

Docs pages https://docs.qoder.com/en/cli/mcp-servers.md and llms.txt index contain **zero timeout keys and zero long-running-tool guidance** — config surface documented is only server add/scopes (`~/.qoder/settings.json`, `${project}/.qoder/settings.local.json`, `${project}/.mcp.json`), types stdio/sse/http/ws, permissions `mcp__<server>__<tool>`. Ground truth below is from the shipped `package/bundle/qodercli.js`.

- **Qoder CLI is a Gemini CLI fork**: bundle ships `@google/gemini-cli-devtools`, Gemini's macOS sandbox `.sb` profiles, and verbatim Gemini `McpClient`/`McpCallableTool` code (identical `progressTokenToCallId`, `registerProgressToken` structure).
- **Per-call timeout: `serverConfig.timeout ?? 600_000 ms` (constant `Y_=6e5`)**. The per-tool wrapper is constructed with `i.timeout ?? Y_` and its `callTool` passes options `{timeout: this.timeout}` — nothing else — to the bundled MCP TypeScript SDK `Client.callTool`. The per-server config key is **`timeout` (milliseconds)** on the `mcpServers` entry (undocumented; the config class fields are `command, args, env, cwd, url, httpUrl, headers, tcp, type, timeout, trust, description, includeTools, excludeTools, plugin, oauth, authProviderType, disabled, alwaysAllow, scope`). `qodercli mcp add` also has a **hidden `--timeout <timeout>` option** (`.hideHelp()`), inherited from `gemini mcp add --timeout`. Max: any JS number (no clamp found). The TS SDK's own default (`bHI=6e4`, 60 s) only applies where no timeout is passed.
- **Progress does NOT reset the timeout.** Qoder generates its own `_meta.progressToken` per call, registers it with a progress reporter, and consumes `notifications/progress` via `setNotificationHandler(...)` → `emitMcpProgress({serverName, callId, progressToken, progress, total, message})` (a UI/SDK event only). It never passes `onprogress`/`resetTimeoutOnProgress`/`maxTotalTimeout` in call options, and the SDK's reset path requires `resetTimeoutOnProgress` (default `false`, verified in the bundle: `I?.resetTimeoutOnProgress??!1`).
- **On timeout: the SDK sends `notifications/cancelled`** (`{requestId, reason}` — verified `cancel()` in the bundled protocol code and in @modelcontextprotocol/sdk 1.23.0 `dist/esm/shared/protocol.js` lines 275-288) and rejects with `McpError -32001 "Request timed out"`. Qoder's wrapper catches and returns to the model `{functionResponse: {name, response: {isError: true, content: [{type:"text", text: "MCP error -32001: Request timed out"}]}}}` — the turn continues.
- **Tasks: SDK-level yes, host-level NO — and a landmine.** The bundled SDK includes the 2025-11-25 Tasks schemas (`taskId`, `status: working|input_required|completed|failed|cancelled`, `ttl`, `pollInterval`, `tasks/get`, `tasks/result`, `tasks/cancel`, `tasks/list`, `execution.taskSupport: required|optional|forbidden`) **and a client-side guard: plain `callTool` THROWS `InvalidRequest` for any tool declaring `taskSupport:"required"`**. Qoder's host code only ever calls plain `callTool` (never task-augmented, never polls `tasks/get`). Consequence: a tool marked task-required is unusable in Qoder; `optional` is called synchronously.
- **Hooks (https://docs.qoder.com/en/cli/hooks.md):** MCP-relevant events: `PreToolUse` / `PostToolUse` / `PostToolUseFailure` / `PermissionRequest` / `PermissionDenied` match MCP tools via matcher (regex `"mcp__.*"` supported); for MCP tools the input includes `mcp_context` (server_name, tool_name, connection info) and `original_request_name`. Dedicated `Elicitation` / `ElicitationResult` events (matcher = `mcp_server_name`; fields `message`, `requested_schema` / `action`, `content`). **No hook fires on MCP progress notifications.** Hook `timeout` fields (command/url default 600 s, prompt 30 s, agent 60 s) are hook-execution timeouts, unrelated to MCP. Notable for long-running designs: hook option `asyncRewake` runs a hook in background and can wake the model with a system reminder on exit code 2 — a possible completion-signal side channel.
- SDK docs (https://docs.qoder.com/en/cli/sdk/mcp.md): `controlRequestTimeoutMs` (default 60_000, 0 disables) is the SDK⇄CLI **control-channel** timeout (elicitation callbacks, mcp* control ops), not the MCP tool-call timeout. Elicitation does **not** pause the tool timer (unlike Codex).

## 3. Gemini CLI (google-gemini/gemini-cli, main @ 0.51.0-nightly.20260625)

- **600 s default confirmed, per-call, configurable per server**: `export const MCP_DEFAULT_TIMEOUT_MSEC = 10 * 60 * 1000;` (`packages/core/src/tools/mcp-client.ts` line 95). Docs: "`timeout` (number): Request timeout in milliseconds (default: 600,000ms = 10 minutes)" per-server in `mcpServers` settings (https://github.com/google-gemini/gemini-cli/blob/main/docs/tools/mcp-server.md, lines ~145 & 171). Applied **per request**: `McpCallableTool.callTool` calls `client.callTool({name, arguments, _meta:{progressToken: randomUUID()}}, undefined, { timeout: this.timeout })` (mcp-client.ts lines ~1440-1461). Same 600 s deadline is used for discovery/refresh operations.
- **No progress-reset**: `resetTimeoutOnProgress` never passed; progress notifications map `progressToken→callId` and are emitted as UI events only.
- **On timeout**: SDK 1.23.0 (`packages/core/package.json`: `"@modelcontextprotocol/sdk": "1.23.0"`) sends `notifications/cancelled` and rejects with `-32001 Request timed out`; the CLI returns `{error: {message, isError: true}}` in the functionResponse to the model.
- **No Tasks**: SDK 1.23.0 predates task support (verified: 0 hits for `isToolTaskRequired`/`tasks/get` in the 1.23.0 npm tarball; tasks landed in SDK ~1.24-1.25, Dec 2025), and no host-side task code.

## Verdict: single-blocking-tool vs start/poll/result

**A single blocking MCP tool call is not viable for minutes-to-hours workflow runs; design the MCP surface as a start/poll/result triad.** Reasons, all source-verified:

1. **No host extends the deadline on progress.** All three hosts hard-cap a tools/call at a fixed wall-clock budget (300 s / 600 s / 600 s default) regardless of `notifications/progress`, even though every host sends a progressToken and two of three SDKs implement reset-on-progress that the hosts simply don't enable. Progress notifications are worth sending for UX (Qoder/Gemini surface them; Codex only logs) but buy zero timeout headroom.
2. **Raising the timeout is possible everywhere but is user-hostile and fragile**: it requires per-user config (`tool_timeout_sec = 86400` / `"timeout": 86400000`), in Qoder the key is undocumented, and a blocking multi-hour call pins the whole agent turn. In Codex, if the deadline still fires, the server is **not** cancelled and the run's handle is lost to the model — orphaned work with no way to reattach. There is also no keepalive/resumption for long-silent tools/call on streamable HTTP in Codex (retries only cover tools/list).
3. **Tasks (2025-11-25) cannot rescue this in any target host today** (July 2026). Codex pins protocol 2025-06-18; Gemini's SDK predates tasks; Qoder's host never issues task-augmented calls. Moreover, declaring `execution.taskSupport: "required"` on tools actively **breaks Qoder** (client-side InvalidRequest before the request is even sent). If task metadata is declared at all, use `"optional"`.
4. **Timeouts are survivable errors** — every host returns an error string into the model's context and continues the turn — so a poll-based design degrades gracefully: even if one poll call times out, the model can call poll again with the same run_id.

Recommended shape: `start_run(...)` → returns `run_id` immediately (<1 s); `get_status(run_id, wait_seconds≤N)` → long-poll variant with `wait_seconds` capped at ~50 s (safely under every default: 60 s worst-case legacy Codex, 300 s current, 600 s others), returning incremental progress/log tail so the model has fresh context each poll; `get_result(run_id)` → final artifacts. Persist run state server-side (survives host restarts and Codex's silent timeout-orphaning). Optionally emit `notifications/progress` during any blocking window for Qoder/Gemini UX. On Qoder, an `asyncRewake` hook can additionally wake the model when a background run finishes.

## UNCERTAINTIES
- Codex official docs (developers.openai.com/codex/mcp) conflict with source: docs say tool_timeout_sec default 60s / startup 10s, but rust-v0.142.5 and main use 300s / 30s. Users on older installed versions (≤v0.98 era) genuinely have 60s defaults; between v0.110-v0.135 it was 120s. Pin behavior to the user's installed Codex version.
- All findings are static source/bundle analysis; no live end-to-end test of an actual >5-minute MCP call was run in any host.
- Codex: behavior when the user interrupts (Esc) mid-MCP-call was not fully traced; based on the architecture (futures dropped, no RequestHandle::cancel call sites found) it almost certainly also skips notifications/cancelled, but this is inferred rather than verified.
- Qoder's per-server 'timeout' (ms) key and hidden 'mcp add --timeout' flag are verified in the 1.0.37 bundle but are undocumented and could change without notice; the exact bundled @modelcontextprotocol/sdk version could not be determined (it is a tasks-aware build, ≥ ~1.24, newer than Gemini's pinned 1.23.0).
- Whether Codex streamable-HTTP transports survive very long silent tools/call windows through proxies/load balancers (SSE keepalive, resumption) was not verified; codex-rs retry logic explicitly covers only tools/list and 404 session recovery.
- Duration::try_from_secs_f64 upper bound (~u64::MAX seconds) is a Rust stdlib guarantee; no additional Codex-side clamp on tool_timeout_sec was found, but an OS/tokio-level practical limit on enormous sleep durations was not tested.
- Qoder docs pages are versionless; qodercli ships near-daily releases (1.0.32-1.0.37 within one week), so bundle-derived behavior may drift quickly.
