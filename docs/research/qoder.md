# Qoder (qoder.com) — Extension Surfaces & Headless Operation (researched 2026-07-03/04)

## 1. What Qoder is today

Qoder is Alibaba's "agentic coding platform" comprising several products (all documented at **docs.qoder.com**; complete machine-readable index at https://docs.qoder.com/llms.txt):

- **Qoder IDE** (desktop app, macOS 11+ / Windows 10-11; https://docs.qoder.com/quick-start.md, https://docs.qoder.com/troubleshooting/common-issue.md). Built on the VS Code platform (third-party evidence: extension version-compat error "Extension is not compatible with Code 1.100.0" in Jimmy Song's Aug 22, 2025 review, https://jimmysong.io/blog/qoder-alibaba-ai-ide-personal-review/; official docs do not state the VS Code lineage or Open VSX usage). Sign-in required (Google/GitHub/email). Two workspaces: **Editor** (collaborative chat/agent) and **Quest** (autonomous delegation).
- **Qoder CLI** — terminal agent, binary **`qodercli`** (launched ~Oct 16, 2025 per https://test-news.aibase.com/news/22048; install `curl -fsSL https://qoder.com/install | bash`, Windows `irm https://qoder.com/install.ps1 | iex`; macOS/Linux/Windows, arm64+amd64, no Windows-on-Arm; docs https://docs.qoder.com/en/cli/quick-start.md). Fully headless-capable (see §4).
- **Agent SDKs** — JS/TS (`@qoder-ai/qoder-agent-sdk` on npm) and Python (`qoder-agent-sdk` on pip, Python 3.10+) that spawn `qodercli` programmatically (https://docs.qoder.com/en/cli/sdk/quick-start.md, https://docs.qoder.com/en/cli/sdk/python/quick-start.md).
- **Cloud Agents** — fully managed REST agent runtime at `https://api.qoder.com/api/v1/cloud/` (https://docs.qoder.com/cloud-agents/overview.md), with a large API surface (agents, sessions, environments, deployments, skills, memory-stores, vaults, files, "dreams" — full endpoint list in llms.txt).
- **JetBrains plugin** (https://docs.qoder.com/plugins/introduction.md) plus sibling consumer products **QoderWork** (desktop general-agent) and **QoderWake** (automation/"Waker" orchestration incl. WakerFlow, https://docs.qoder.com/qoderwake/wakerflow.md) and a **mobile app/web** for remote control (https://docs.qoder.com/mobile/app/remote-control.md).
- **Quest Mode**: agent-first delegation workspace — "Define the goal. Review the result." Single-agent **Agent mode** or multi-agent parallel **Experts mode**; kanban-style "My Quests" board with statuses Running/Waiting/Completed; task pin/fork/rename; tiled multi-pane monitoring (https://docs.qoder.com/user-guide/quest/overview.md).
- Models: Qwen family prominently (e.g., "Qwen3.7-Max" promos, https://docs.qoder.com/events/qwen-max-daily-free.md), auto model selection ("unified intelligence"), custom models page https://docs.qoder.com/user-guide/chat/custom-models.md. Knowledge Engine (Repo Wiki auto-generated internal index, memory, knowledge cards): https://docs.qoder.com/user-guide/repo-wiki.md, https://docs.qoder.com/user-guide/knowledge-engine/memory.md.

## 2. Extension surfaces (the model is heavily Claude-Code-convergent: `.qoder/` dir, AGENTS.md, SKILL.md, hooks JSON, `mcp__server__tool` naming)

### MCP
- **IDE**: Settings → MCP → "My Servers" → + Add (JSON) or install from **MCP Square** gallery. `mcpServers` JSON object; STDIO (`command`, `args`, `env`) and SSE/Streamable HTTP (`type: "sse"`, `url`; streamable HTTP auto-detected). Per-tool confirmation prompt with "auto-run subsequent" option. (https://docs.qoder.com/user-guide/chat/model-context-protocol.md)
- **CLI** (https://docs.qoder.com/en/cli/mcp-servers.md): config tiers `~/.qoder/settings.json` (user), `${project}/.qoder/settings.local.json` (local), `${project}/.mcp.json` (project, committed). Subcommands `qodercli mcp add <name> -- <command>`, `mcp list`, `mcp remove`; `/mcp reload` in-session. Scope flag `-s user|local|project` (default local); transport flag `-t stdio|sse|http|ws`. Tool permission naming `mcp__<server>__<tool>` with wildcards (`"allow": ["mcp__context7__*"]`).

### Rules / memories (`.qoder/` directory)
- **IDE Rules**: `.qoder/rules/*.md` in project (commit to share); 4 trigger types — Always Apply, Model Decision, Specific Files (glob like `src/**/*.ts`), Apply Manually (`@rule`); 100k-char cap across active rules; **AGENTS.md in project root is auto-recognized** (https://docs.qoder.com/user-guide/rules.md).
- **CLI Memory** (https://docs.qoder.com/en/cli/memory.md): `~/.qoder/AGENTS.md` (user), `<project>/AGENTS.md` (team, commit), `<project>/AGENTS.local.md` (machine-local), `.qoder/rules/*.md` with frontmatter `trigger: always_on | manual | model_decision (+description) | glob`. `@path/to/file` import syntax (relative/absolute/`~/`; ignored in code blocks; external imports need approval). `/init` bootstraps, `/memory` and `/memory manage` manage. Auto-memory opt-in via env `QODER_MEMORY=1` → `~/.qoder/projects/<project>/memory/`; user scope via `QODER_MEMORY_USER=1` → `~/.qoder/memory/`. Upward walk stops at `.git`; subdirectory AGENTS.md loads on-demand.

### Skills — SKILL.md standard: effectively yes
- Both IDE and CLI: `~/.qoder/skills/{name}/SKILL.md` (user) and `.qoder/skills/{name}/SKILL.md` (project; project overrides user). YAML frontmatter `name` (lowercase/digits/hyphens, ≤64 chars) + `description` (≤1024 chars); optional auxiliary files (REFERENCE.md, scripts/, templates/) referenced for progressive disclosure. Triggered automatically by model matching on description, or manually via `/skill-name`. (CLI: https://docs.qoder.com/en/cli/Skills.md; IDE: https://docs.qoder.com/extensions/skills.md)
- The IDE Skills page says skills "work identically in both Qoder IDE and CLI" and documents installing from the **skills.sh marketplace with `npx skills add <repo> -a qoder`** — i.e., the ecosystem's cross-agent Agent-Skills installer explicitly supports Qoder as a target; format is byte-compatible with Anthropic's Agent Skills SKILL.md. Docs never name-check "Anthropic Agent Skills standard" explicitly. Built-in skills: `/create-skill`, `/canvas`, `/vercel-deploy`.

### Custom agents / subagents
- **CLI subagents** (https://docs.qoder.com/en/cli/subagent.md): Markdown + YAML frontmatter at `.qoder/agents/<name>.md` (project) / `~/.qoder/agents/<name>.md` (user) / `--agents '<JSON>'` (ephemeral flag). Fields: `name`, `description` (required); `tools` (array or CSV; supports `Agent(Explore, Plan)` to restrict nested dispatch, `mcp__server__tool` names, wildcards), `disallowedTools`, `model` (default `inherit`), `maxTurns`, `timeoutMins`, `permissionMode` (`default|acceptEdits|dontAsk|bypassPermissions|auto|plan`), `mcpServers` (reference or inline with `include_tools`), `isolation: worktree`, `background: true`, `initialPrompt`. Priority (low→high): built-in → user → project → plugin → `--agents` flag; `qodercli agents list` shows shadowing. Invocation: natural language, `@agent-name` mention, implicit description matching, or **as main agent: `qodercli --agent api-reviewer -p "..."`**. Built-ins: `general-purpose`, `Explore`, `Plan`, `qoder-guide`, `statusline-setup`, `SaveMemory`. `settings.json` `agents.overrides` can retune tools/maxTurns/temperature without editing files. Creation wizard: `/agents` TUI.
- **IDE Custom Agents** (https://docs.qoder.com/extensions/subagent.md): same file locations/format; fields `name`, `description`, `model`, `tools` (Bash, Edit, Write, Glob, Grep, Read, WebFetch, WebSearch), `skills`, `mcpServers`; create via `/create-agent <requirement>`; invoke via `/agent-name` or automatic intent matching.

### Custom commands
- **CLI** (https://docs.qoder.com/en/cli/command.md): `.qoder/commands/<name>.md` (project) / `~/.qoder/commands/<name>.md` (user); Markdown + frontmatter (`name`, `description`); subdirectories namespace with `:` (`commands/git/commit.md` → `/git:commit`); a dir containing SKILL.md registers as a single command. Built-in slash commands include `/agents /tasks /workflows /clear /commands /config /help /init /login /logout /model /status /vim` and prompt-level `/compact /quest /review` usable headlessly: `qodercli -p '/review Focus on security vulnerabilities'`. Docs don't document `$ARGUMENTS`-style interpolation. IDE commands: https://docs.qoder.com/user-guide/commands.md.

### Hooks
- **CLI** (https://docs.qoder.com/en/cli/hooks.md): 24+ events — SessionStart/End, PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest, PermissionDenied, Stop, StopFailure, SubagentStart/Stop, PreCompact/PostCompact, InstructionsLoaded, ConfigChange, CwdChanged, FileChanged, WorktreeCreate/Remove, UserPromptSubmit, Notification, Elicitation/ElicitationResult. Config merged from `~/.qoder/settings.json`, `.qoder/settings.json`, `.qoder/settings.local.json`. Four hook types: `command`, `http` (POST JSON to URL), `prompt` (isolated LLM call), `agent` (spawn verification subagent). Matchers: exact, `|`-alternation, regex, `*`; per-hook `if` filter with glob on primary args. stdin JSON (`session_id`, `transcript_path`, `cwd`, `hook_event_name`, `permission_mode`, `agent_id`, `agent_type` + event fields); exit 0 = ok (stdout parsed), exit 2 = block (stderr fed to agent); output JSON supports `continue`, `decision: allow|deny`, `permissionDecision`, `updatedInput`, `additionalContext`, `hookSpecificOutput`. Env vars in hook process: `QODER_PROJECT_DIR`, `QODER_PLUGIN_ROOT`, `QODER_PLUGIN_DATA`.
- **IDE/JetBrains hooks** (https://docs.qoder.com/extensions/hooks.md): subset — UserPromptSubmit, PreToolUse, PostToolUse, PostToolUseFailure, Stop; same config files ("IDE/JB plugin and CLI share the same config files"); `type` must be `"command"`; no hot reload (restart IDE); input includes git context `extra.email/repo/branch`.

### Plugins
- **CLI plugins** (https://docs.qoder.com/en/cli/plugins.md): directory bundle with `.qoder-plugin/plugin.json` manifest (only `name` required; `version`, `description`, `author`, `homepage`, `repository`, `license`, `keywords`) containing `commands/`, `agents/`, `skills/`, `hooks/hooks.json`, `output-styles/`, `bin/`, `.mcp.json`. Commands: `qodercli plugins install <path|name> [--scope user|project|local]`, `uninstall`, `enable/disable`, `list [--json]`, `validate`, `/plugins reload`. **Marketplace system**: `plugins marketplace add <git-repo|shorthand|path|url>`, plugin IDs `name@marketplace-name`; state tracked in `settings.json` `enabledPlugins`.
- **IDE plugins** (https://docs.qoder.com/extensions/plugins.md): Marketplace in-IDE bundling Skills + Commands + MCP servers; "Plugins are currently only available in Quest". Separate from any VS Code extension mechanism (docs are silent on VS Code extension compatibility/Open VSX).
- Enterprise marketplace controls: https://docs.qoder.com/account/enterprise/marketplace.md.

## 3. Built-in workflow/orchestration features
- **Quest Mode**: spec-driven flow (toggle **Spec** on task creation → multiple-choice requirement clarification → generated spec doc with requirement description, design plan, task breakdown, acceptance criteria → annotate/modify → **Build** → live to-do-list progress → Review panel with stage/commit/push) (https://docs.qoder.com/user-guide/quest/spec-driven.md). **Goal mode** = iterate autonomously until objective met (https://docs.qoder.com/user-guide/quest/goal-driven.md). **Experts mode** = multiple agents in parallel (https://docs.qoder.com/user-guide/quest/experts-mode.md). **Scheduled tasks** via `/schedule` or natural-language timing; runs if Qoder is open (https://docs.qoder.com/user-guide/quest/scheduled-tasks.md). **Execution environments**: Local vs **Worktree** (isolated git worktree per task, parallel tasks, "Move to local" merge-back, startup scripts, 15-worktree retention default) (https://docs.qoder.com/user-guide/quest/execution-environments.md).
- **CLI Dynamic Workflows** (https://docs.qoder.com/en/cli/workflows.md): saved **JavaScript** files in `.qoder/workflows` (project) / `~/.qoder/workflows` (user) / plugins / built-ins; `export const meta = {name, description, phases:[{title, detail}]}`; orchestration helpers `agent()`, `parallel()`, `pipeline()`, `phase()`, `log()`, `args`; background multi-agent execution monitored via `/workflows`; scripts sandboxed (no direct fs/network/shell — side effects only through child agents under permissions/hooks).
- **CLI**: `/quest` prompt command; `--worktree` flag for parallel isolated sessions; sequential/parallel subagent orchestration in prompts; **Goal Mode** `/goal set <objective>` auto-switches to `auto` permission mode (https://docs.qoder.com/en/cli/permissions.md).
- **QoderWake WakerFlow** = separate workflow orchestration product (https://docs.qoder.com/qoderwake/wakerflow.md).

## 4. Headless / programmatic operation — YES, first-class
- **Print mode**: `qodercli -p "prompt"` / `--print`; `--output-format text|json|stream-json`; `-w <workspace>`, `-c` (continue), `-r <session-id>` (resume), `--max-turns`, `--allowed-tools`/`--disallowed-tools`, `--permission-mode default|accept_edits|auto|bypass_permissions|dont_ask` (plus `plan`), `--yolo`, `--worktree`, `--add-dir`, `--settings <path>`, `--agents <json>`, `--agent <name>` (https://docs.qoder.com/en/cli/using-cli.md, https://docs.qoder.com/en/cli/permissions.md). In headless, `ask` permission decisions auto-deny; `dont_ask` is the explicit headless mode. Headless auth: env `QODER_PERSONAL_ACCESS_TOKEN` (PAT from https://qoder.com/account/integrations).
- **Agent SDKs**: JS `@qoder-ai/qoder-agent-sdk` — `query({prompt, options})` returns `AsyncGenerator<SDKMessage>`; message types `system|assistant|user|result|stream_event|hook_event`; options incl. `auth: accessTokenFromEnv()`, `allowedTools`, `permissionMode`, `includePartialMessages` (stream deltas: `text_delta`, `thinking_delta`, `input_json_delta`), `canUseTool` permission callback, hooks callbacks, model policy provider (https://docs.qoder.com/en/cli/sdk/quick-start.md, https://docs.qoder.com/en/cli/sdk/streaming-output.md, https://docs.qoder.com/en/cli/sdk/references). Python `qoder-agent-sdk`: `query()`, `QoderSDKClient` (multi-turn), `QoderAgentOptions`, `AssistantMessage/ResultMessage/TextBlock/ToolUseBlock` (https://docs.qoder.com/en/cli/sdk/python/quick-start.md). SDK spawns qodercli under the hood. SDK also has sessions, checkpoint/rewind, plugins, skills, subagents, MCP, cloud-agent (experimental) modules.
- **ACP**: `qodercli --acp` runs as an Agent Client Protocol server over stdio — documented Zed integration via `agent_servers` in `~/.config/zed/settings.json` (https://docs.qoder.com/en/cli/acp.md).
- **Cloud/remote**: `qodercli --remote "task"` executes on Qoder-managed cloud VMs (streams back; Ctrl+C detaches, task continues; `/remote-env` picks environment; parallel remote sessions; monitored at Cloud Agents Console) (https://docs.qoder.com/en/cli/cloud-mode.md). `qodercli remote-control` = daemon mode receiving tasks from mobile/web (https://qoder.com/agents); `/remote-control` pairs an existing session via QR (https://docs.qoder.com/en/cli/remote-control.md).
- **CI**: GitHub Action **`QoderAI/qoder-action@v0`** — inputs `qoder_personal_access_token`, `prompt` (e.g. `/review-pr REPO:... PR_NUMBER:...`); qoderai GitHub App via integrations.qoder.com; respects repo AGENTS.md (https://docs.qoder.com/en/cli/qoder-action.md).
- **Cloud Agents REST API**: base `https://api.qoder.com/api/v1/cloud/`, `Authorization: Bearer <PAT>`; define Agent (model+system prompt+tools) → Environment (container runtime) → Session → send `user.message` → stream events via SSE or poll (https://docs.qoder.com/cloud-agents/overview.md; endpoint pages under https://docs.qoder.com/cloud-agents/api/...; also Teams OpenAPI for admin metrics https://docs.qoder.com/account/teams/openapi/index.md; archived OpenAPI spec https://docs.qoder.com/archived/api-reference/openapi.json).

## Practical integration takeaways for an orchestrator
1. Headless driving: `QODER_PERSONAL_ACCESS_TOKEN=... qodercli -p "<task>" --output-format stream-json --permission-mode dont_ask|auto --max-turns N -w <dir> [--worktree]`; resume with `-r <session-id>`; or use the JS/Python Agent SDK for typed events + permission callbacks.
2. Qoder can also be a spawnee via ACP (`qodercli --acp`) — any ACP client can host it.
3. Third-party extension drop-in points inside a repo: `AGENTS.md`, `.qoder/rules/*.md`, `.qoder/skills/*/SKILL.md`, `.qoder/agents/*.md`, `.qoder/commands/*.md`, `.qoder/workflows/*.js`, `.qoder/settings.json` (hooks/permissions/agents overrides), `.mcp.json` — all git-committable and shared between CLI and (for skills/agents/rules/hooks) the IDE.
4. Anthropic-style SKILL.md skills work as-is; skills.sh installer targets Qoder (`npx skills add <owner/repo> -a qoder`).

## KEY FACTS
- Qoder CLI binary is `qodercli`; install via `curl -fsSL https://qoder.com/install | bash` (macOS/Linux) or `irm https://qoder.com/install.ps1 | iex` (Windows); supports macOS/Linux/Windows on arm64/amd64 [https://docs.qoder.com/en/cli/quick-start.md]
- Headless print mode: `qodercli -p/--print` with `--output-format text|json|stream-json`, plus `-w` workspace, `-c` continue, `-r <session-id>` resume, `--allowed-tools`, `--disallowed-tools`, `--max-turns`, `--yolo`, `--worktree` [https://docs.qoder.com/en/cli/using-cli.md]
- Headless auth via env var `QODER_PERSONAL_ACCESS_TOKEN` (PAT generated at qoder.com/account/integrations); `/login`-provided token takes precedence if both set [https://docs.qoder.com/en/cli/quick-start.md]
- Permission modes: default, accept_edits, auto (AI classifier), bypass_permissions (--yolo), dont_ask (headless auto-deny), plus plan mode and Goal Mode (`/goal set <objective>` auto-switches to auto); rules syntax `ToolName(pattern)` under permissions.allow/ask/deny in settings.json; headless environments auto-deny `ask` decisions [https://docs.qoder.com/en/cli/permissions.md]
- MCP config tiers for CLI: `~/.qoder/settings.json` (user), `.qoder/settings.local.json` (local), `${project}/.mcp.json` (project/committed); `qodercli mcp add <name> -- <cmd>` with `-s user|local|project` and `-t stdio|sse|http|ws`; MCP tool permission naming `mcp__<server>__<tool>` [https://docs.qoder.com/en/cli/mcp-servers.md]
- Skills use SKILL.md with YAML frontmatter (name ≤64 chars, description ≤1024 chars) at `~/.qoder/skills/{name}/SKILL.md` and `.qoder/skills/{name}/SKILL.md`; project overrides user; auto-trigger by description matching or manual `/skill-name`; identical in IDE and CLI; installable from skills.sh via `npx skills add <repo> -a qoder` [https://docs.qoder.com/en/cli/Skills.md and https://docs.qoder.com/extensions/skills.md]
- Subagents are Markdown+YAML at `.qoder/agents/<name>.md` / `~/.qoder/agents/<name>.md` with fields name, description, tools, disallowedTools, model, maxTurns, timeoutMins, permissionMode, mcpServers, isolation: worktree, background; run as main agent via `qodercli --agent <name> -p "..."` or inject ephemeral agents via `--agents '<JSON>'` [https://docs.qoder.com/en/cli/subagent.md]
- CLI hooks support 24+ events (PreToolUse, PostToolUse, Stop, SubagentStart/Stop, UserPromptSubmit, WorktreeCreate, Elicitation, etc.) with four hook types: command, http, prompt, agent; exit 2 blocks with stderr fed to the agent; config merged from ~/.qoder/settings.json, .qoder/settings.json, .qoder/settings.local.json [https://docs.qoder.com/en/cli/hooks.md]
- Memory: AGENTS.md at `~/.qoder/AGENTS.md`, `<project>/AGENTS.md`, `<project>/AGENTS.local.md` plus `.qoder/rules/*.md` with frontmatter trigger: always_on|manual|model_decision|glob; `@path` import syntax; auto-memory opt-in via QODER_MEMORY=1 [https://docs.qoder.com/en/cli/memory.md]
- Agent SDKs: npm `@qoder-ai/qoder-agent-sdk` and pip `qoder-agent-sdk` (Python 3.10+); `query()` returns async iterable of SDKMessage (system/assistant/user/result/stream_event/hook_event); `QoderSDKClient` for multi-turn; `includePartialMessages` yields text_delta/thinking_delta/input_json_delta; SDK spawns qodercli internally [https://docs.qoder.com/en/cli/sdk/quick-start.md, https://docs.qoder.com/en/cli/sdk/python/quick-start.md, https://docs.qoder.com/en/cli/sdk/references]
- `qodercli --acp` runs the CLI as an Agent Client Protocol server over stdio; documented Zed editor integration via agent_servers config [https://docs.qoder.com/en/cli/acp.md]
- CLI plugins: directory with `.qoder-plugin/plugin.json` manifest bundling commands/, agents/, skills/, hooks/hooks.json, output-styles/, bin/, .mcp.json; `qodercli plugins install <path> --scope user|project|local`; marketplace system with IDs `name@marketplace-name` [https://docs.qoder.com/en/cli/plugins.md]
- Dynamic Workflows: JavaScript files in `.qoder/workflows` / `~/.qoder/workflows` with `export const meta = {name, description, phases}`; helpers agent(), parallel(), pipeline(), phase(), log(), args; sandboxed (no direct fs/shell/network); monitored via /workflows [https://docs.qoder.com/en/cli/workflows.md]
- Quest Mode spec-driven flow: Spec toggle → requirement clarification questions → generated spec (requirements, design plan, task breakdown, acceptance criteria) → Build → live to-do progress → Review/commit; Goal toggle for autonomous iteration; /schedule for timed tasks; Local vs Worktree execution environments with parallel isolated worktrees (15 retained by default) [https://docs.qoder.com/user-guide/quest/spec-driven.md, https://docs.qoder.com/user-guide/quest/execution-environments.md]
- Cloud execution: `qodercli --remote "task"` runs on Qoder-managed cloud VMs (detachable, parallel); `qodercli remote-control` daemon accepts tasks from mobile/web at qoder.com/agents; Cloud Agents REST API at https://api.qoder.com/api/v1/cloud/ with Bearer PAT auth (agents, environments, sessions, SSE event streaming) [https://docs.qoder.com/en/cli/cloud-mode.md, https://docs.qoder.com/en/cli/remote-control.md, https://docs.qoder.com/cloud-agents/overview.md]
- GitHub Action `QoderAI/qoder-action@v0` with inputs qoder_personal_access_token and prompt (e.g. /review-pr); requires qoderai GitHub App from integrations.qoder.com [https://docs.qoder.com/en/cli/qoder-action.md]

- China edition: Lingma rebranded to Qoder CN; CLI binary `qoderclicn` with same model (print mode text/json/stream-json, MCP via .qoder-cn.json + .mcp.json, agents in ~/.qoder-cn/agents/, AGENTS.md memory) [https://help.aliyun.com/en/lingma/qodercli-cn/user-guide/using-the-cli and https://plugins.jetbrains.com/plugin/17809-qoder-cn-formerly-lingma--alibaba-cloud-ai-coding-assistant]
- Full docs index is machine-readable at https://docs.qoder.com/llms.txt and every docs page is fetchable as raw markdown by appending .md [https://docs.qoder.com/llms.txt]

A search snippet claimed 'Qoder 1.0 released May 15, 2026 for Windows/macOS/Linux' but this is weakly sourced (baike.baidu.com via search summary) and unverified; docs still describe the IDE as macOS 11+/Windows 10-11 only, so 'Linux IDE support' is doubtful.
- VS Code lineage of Qoder IDE: official docs never state it is a VS Code fork or which extension marketplace (Open VSX?) it uses; the strongest evidence is a version-compat error string in a third-party review (jimmysong.io, Aug 2025). VS Code extension compatibility should be treated as probable-but-unconfirmed.
- Agent Skills standard compliance: SKILL.md format matches Anthropic's Agent Skills spec field-for-field and skills.sh's installer targets qoder (`npx skills add ... -a qoder`), but Qoder docs never explicitly claim compatibility with the Anthropic standard.
- Permission mode naming is inconsistent across doc pages: the permissions page uses snake_case (accept_edits, bypass_permissions, dont_ask) while the subagent page uses camelCase frontmatter values (acceptEdits, dontAsk, bypassPermissions); likely CLI flag vs frontmatter conventions, unverified.
- Cloud Agents API scope: llms.txt lists endpoints for deployments, dreams, skills, vaults, memory-stores, files and work-queues, but the overview page (as summarized) only describes agents/environments/sessions/events as core resources — the extra resources exist per the endpoint index but their semantics (especially 'Dreams') were not investigated.
- Custom command argument interpolation ($ARGUMENTS-style) is not documented on the command page; unknown whether arguments are passed positionally or via prompt text.
- One fetched summary of the Quest overview page contained the phrase 'Anthropic Qoder' — this is an artifact of the summarizer model, not the source; Qoder is Alibaba's product.
- The exact JSON schema of `--output-format json` / `stream-json` CLI output (top-level envelope fields, result subtypes, cost/usage fields) is not fully documented on fetched pages; SDK references page gives the SDKMessage union but the raw CLI stream-json event reference may differ; a forum thread (forum.qoder.com/t/qoder-cli-p-output-format-stream-json/9599) suggests stream-json for -p was a user-requested feature added after initial release.
- All page contents were extracted via a summarizing fetch tool; minor field-name paraphrase errors are possible — before implementation, re-fetch the raw .md pages (append .md to any docs.qoder.com path) for authoritative syntax.
