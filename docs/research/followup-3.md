# GAP: codex exec --output-schema enforcement mechanics, failure mode, and exit-code semantics

# codex exec --output-schema: enforcement mechanics, failure modes, exit codes

All source references are pinned to tag `rust-v0.142.4` (installed binary on this machine is `codex-cli 0.142.4`; latest release is `rust-v0.142.5`, published 2026-07-01: https://github.com/openai/codex/releases/tag/rust-v0.142.5). An empirical run of `codex exec` with an impossible schema was NOT performed (this session is a read-only planning phase; running exec writes `~/.codex` session state and calls the API). Behavior is instead pinned by the repo's own integration tests, which assert exact request bodies and exit codes.

## 1. Enforcement layer: 100% server-side (OpenAI Responses API structured outputs, strict mode). Zero local validation.

Full data path:

1. **CLI flag**: `--output-schema <FILE>` — "Path to a JSON Schema file describing the model's final response shape", `global = true` so it also applies to the `resume` subcommand. (`codex-rs/exec/src/cli.rs` lines 52-54, https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/cli.rs)
2. **Client-side loading** — `load_output_schema()` in `codex-rs/exec/src/lib.rs` (~line 1787): reads the file and parses it with `serde_json::from_str::<Value>` **only**. The only client-side checks are "file readable" and "is valid JSON". It is NOT checked to be a valid JSON Schema, NOT checked against the structured-outputs strict subset. Failure of either check → message on stderr (`Failed to read output schema file ...` / `Output schema file ... is not valid JSON: ...`) → `std::process::exit(1)` before any API call. (https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/lib.rs)
3. **Transport to core**: the parsed `Value` is placed verbatim in `TurnStartParams.output_schema` on the JSON-RPC `turn/start` request to the *in-process* app-server (exec no longer talks to core directly; it embeds an app-server client). The app-server maps it to `Op::UserInput { final_output_json_schema: params.output_schema, ... }` (`codex-rs/app-server/src/request_processors/turn_processor.rs` ~line 523).
4. **Attachment to model requests**: core stores it as `TurnContext.final_output_json_schema` and `build_prompt()` copies it into **every** sampling `Prompt` of the turn: `output_schema: turn_context.final_output_json_schema.clone(), output_schema_strict: !is_guardian_reviewer_source(...)` — i.e. `strict` is always `true` for exec sessions (only internal "guardian reviewer" sessions get non-strict). (`codex-rs/core/src/session/turn.rs` lines 1104-1113, https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/core/src/session/turn.rs)
5. **Wire format**: `create_text_param_for_request()` (`codex-rs/codex-api/src/common.rs` lines 310-328) builds the Responses API `text` param; core attaches it in `codex-rs/core/src/client.rs` (~line 816). Exact request JSON, asserted byte-for-byte by the integration test `exec_includes_output_schema_in_request` (`codex-rs/exec/tests/suite/output_schema.rs`):

```json
"text": { "format": {
    "name": "codex_output_schema",
    "type": "json_schema",
    "strict": true,
    "schema": <user schema verbatim, no normalization>
}}
```
(https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/tests/suite/output_schema.rs, https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/codex-api/src/common.rs)

6. **No local output validation**: the workspace has no JSON-Schema *validator* crate at all (only `schemars 0.8.22` for config-schema generation; `grep jsonschema codex-rs/Cargo.toml` → nothing; no `validate_output_schema` symbol repo-wide). The final `agent_message` text is passed through as an opaque string into events and the `-o` file — never parsed, never validated. The exec event doc comment says: "Either a natural-language response or a JSON string when structured output is requested" (`codex-rs/exec/src/exec_events.rs`).
7. **Wire API note**: as of 0.142.x, `WireApi` has exactly one variant, `Responses` — the Chat Completions wire API was removed (`codex-rs/model-provider-info/src/lib.rs` lines 53-60). So there is no separate chat-completions `response_format` path; everything (including `--oss` lmstudio/ollama providers) goes through Responses-shaped requests. Whether a local OSS server actually *enforces* `text.format` is up to that server (see uncertainties).

**Consequences of strict:true server enforcement**: when the request is accepted, the model output is produced by constrained decoding — it *will* parse as JSON conforming to the schema (exceptions: token-limit truncation, refusals). "The model couldn't satisfy the schema" is therefore not a runtime failure mode; the real failure mode is **the schema itself being rejected up front with HTTP 400** because it violates the structured-outputs subset (root must be `type:"object"`, all properties `required`, `additionalProperties:false`, no unsupported keywords, etc.). A `{"const": false}`-style impossible schema would be rejected the same way (400 `invalid_json_schema`), not looped on.

## 2. Error path when the API rejects the schema (HTTP 400)

Chain, fully traced in source:

1. POST `/v1/responses` returns 400 → `TransportError::Http { status: 400, body }` → `map_api_error()` (`codex-rs/codex-api/src/api_bridge.rs` lines 63-82): status `BAD_REQUEST` maps to `CodexErr::InvalidRequest(body_text)` (unless it's the cyber-policy code or the invalid-image string).
2. `CodexErr::InvalidRequest` has `is_retryable() == false` (`codex-rs/protocol/src/error.rs` lines 173-210) → the sampling retry loop in `run_sampling_request` (`codex-rs/core/src/session/turn.rs` ~line 1204: `if !err.is_retryable() { return Err(err); }`) returns immediately — **0 retries** for a bad schema; the failure is deterministic and instant.
3. The turn driver's generic error arm (`turn.rs` ~lines 456-465) emits `EventMsg::Error` and breaks the turn.
4. App-server (`codex-rs/app-server/src/bespoke_event_handling.rs`): `EventMsg::Error` → `handle_error_notification` → `ServerNotification::Error { error, will_retry: false, thread_id, turn_id }` AND stores `turn_summary.last_error`. When the turn completes, `handle_turn_complete` (lines 1523-1550) computes `(TurnStatus::Failed, Some(error))` because `last_error` is set → `TurnCompleted` notification with `status: Failed`.
5. exec (`codex-rs/exec/src/lib.rs` run loop ~lines 985-1005): sets `error_seen = true` on (a) `ServerNotification::Error` with `will_retry == false` for the primary thread/turn, or (b) `TurnCompleted` with status `Failed` or `Interrupted`. After the event loop and `print_final_output()`, `if error_seen { std::process::exit(1); }`.

**Observed JSONL (`--json`) on a schema-rejected run** (from event mapping in `codex-rs/exec/src/event_processor_with_jsonl_output.rs` lines 438-449 and 515-548, and the exact error payload reported in issue #16552):

```jsonl
{"type":"thread.started","thread_id":"<uuid>"}
{"type":"turn.started"}
{"type":"error","message":"... {\"type\":\"invalid_request_error\",\"code\":\"invalid_json_schema\",\"message\":\"Invalid schema for response_format 'codex_output_schema': schema must be a JSON Schema of 'type: \\\"object\\\"', got 'type: \\\"array\\\"'.\",\"param\":\"text.format.schema\"} ..."}
{"type":"turn.failed","error":{"message":"..."}}
```
Exit code 1. On `turn.failed` the processor clears `final_message` and sets `emit_final_message_on_shutdown=false`, so the `-o/--output-last-message` file is **not written at all** (not even truncated — a stale file from a previous run survives). On `TurnStatus::Interrupted` (Ctrl-C): no `turn.failed` event is emitted (just shutdown), but exit code is still 1.

**Retryable vs fatal `error` events — critical for consumers**: `EventMsg::StreamError` (transient stream drops, 5xx, timeouts, 429-without-usage-limit) is forwarded as `ServerNotification::Error { will_retry: true }` (bespoke_event_handling.rs lines 982-998) and the JSONL processor maps **both** retryable and fatal notifications to the same `{"type":"error","message":...}` event (it does not expose `will_retry`, messages look like `Reconnecting... 2/5`). So a JSONL `error` event alone does NOT mean the run failed. `turn.failed` and the process exit code are the only authoritative failure signals.

**Internal retry policy for retryable errors**: retryable `CodexErr` variants (`Stream`, `Timeout`, `RequestTimeout`, `UnexpectedStatus` (!), `ResponseStreamFailed`, `ConnectionFailed`, `InternalServerError`, `Io`, `Json`, ...) are retried by `handle_retryable_response_stream_error` (`codex-rs/core/src/responses_retry.rs`) up to `provider.stream_max_retries()` times with exponential backoff (server-requested delay honored for `Stream` errors), plus one WebSocket→HTTPS transport fallback that resets the counter. Defaults (`codex-rs/model-provider-info/src/lib.rs` lines 26-33): `DEFAULT_STREAM_MAX_RETRIES = 5`, `DEFAULT_REQUEST_MAX_RETRIES = 4`, user-configurable per provider in config.toml (`model_providers.<id>.stream_max_retries`, hard cap 100). Non-retryable (immediate turn failure): `InvalidRequest` (HTTP 400 = bad schema), `UsageLimitReached` (429 usage_limit_reached), `QuotaExceeded`, `RefreshTokenFailed`, `ContextWindowExceeded` (after auto-compaction attempts), `CyberPolicy`, `Sandbox`, `Fatal`, `ServerOverloaded`.

## 3. Exit-code table for `codex exec` (v0.142.4)

The docs only say "non-zero"; source shows there is effectively **one** failure code:

| Code | Cause | Source |
|---|---|---|
| 0 | Turn completed (`TurnStatus::Completed`), no non-retryable error notification seen. Includes runs where sandbox denied commands, approvals were auto-rejected, or the model gave a "wrong" answer — those don't fail the turn. | lib.rs run loop |
| 1 | **Everything fatal.** (a) `turn.failed` (schema 400, retry exhaustion, usage limit, quota, context-window, auth/refresh failure at request time, `response.failed` from server); (b) turn interrupted (Ctrl-C → `turn/interrupt` → `TurnStatus::Interrupted`); (c) non-retryable `Error` notification; (d) pre-flight exits: unreadable/invalid `--output-schema` file, bad `-c` override parse, config.toml load error, execpolicy rules error, login-restriction violation, not-a-git-repo without `--skip-git-repo-check`, no prompt on stdin, invalid UTF-8 prompt, codex-home resolution failure, required MCP server failed to init; (e) any anyhow error bubbling out of `run_main` (e.g., app-server client init failure, `turn/start` RPC error) — Rust's `Termination` for `Result` prints `Error: ...` to stderr and exits 1. | `std::process::exit(1)` at lib.rs lines 306, 324, 475, 494, 665, 785, 1049, 1797, 1808, 1894, 1909, 1916, 1925; tests `server_error_exit.rs` (`.code(1)` on `response.failed`), `mcp_required_exit.rs` (`.code(1)`) |
| 2 | CLI usage error (unknown flag, bad value) — clap's default. | `TopCli::parse()` in main.rs |
| 101 | Rust panic (should not normally happen). | Rust runtime default |

There are no distinct codes for auth vs sandbox vs schema failures. Sandbox denials specifically do NOT affect the exit code: in exec mode all approval requests (`ExecCommandApproval`, `ApplyPatchApproval`, `PermissionsRequestApproval`, elicitations) are auto-rejected via `reject_server_request` (lib.rs ~1744-1780) and the command failure is just fed back to the model; `error_seen` is only set if the JSON-RPC reject itself errors.

Sources: https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/lib.rs, https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/tests/suite/server_error_exit.rs, https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/tests/suite/mcp_required_exit.rs

## 4. GitHub issues cross-check

- **#16552** (open, 2026-04-02): exact production repro of the 400 path — `invalid_request_error` / `invalid_json_schema`, `param: "text.format.schema"`, "Invalid schema for response_format 'codex_output_schema': schema must be a JSON Schema of 'type: \"object\"', got 'type: \"array\"'". Deterministic, fires on every request. https://github.com/openai/codex/issues/16552
- **#19816** (open, 2026-04-27): `--output-schema` does NOT apply only to final output — because `final_output_json_schema` is copied into every sampling prompt, **intermediate progress `agent_message` items are also schema-shaped JSON** (e.g. `{"final": "I will run pwd..."}`). Consumers that grab the *first* schema-valid agent_message get wrong results. Must take the LAST one. https://github.com/openai/codex/issues/19816
- **#15451** (closed, 2026-03-22): older report that `--json`/`--output-schema` were "silently ignored when tools/MCP servers are active" (title; closed — behavior in 0.142.4 attaches the schema regardless of tools). https://github.com/openai/codex/issues/15451

## 5. Recommended agent()-level retry contract for Codex

1. **Pre-validate the schema locally before ever invoking codex** (root `type:"object"`, every property in `required`, `additionalProperties:false`, structured-outputs-supported keywords only). Codex sends it verbatim with `strict:true`; an incompatible schema fails deterministically (400, zero internal retries, exit 1) on *every* attempt — retrying at the agent level is pure waste. Classify by grepping the `turn.failed`/`error` message for `invalid_json_schema` / `Invalid schema for response_format 'codex_output_schema'` → map straight to null/permanent-failure.
2. **Success signal**: exit code 0 AND a `turn.completed` event. Treat exit 1 + `turn.failed` as failure; treat exit 1 without `turn.failed` as interrupted/infra failure. Ignore standalone `{"type":"error"}` JSONL events — they include benign `Reconnecting... n/5` retry notices.
3. **Result extraction**: use the LAST `item.completed` event with `"type":"agent_message"` (or equivalently the `-o` file, which is exactly the last agent message and is only written on `turn.completed`). Never the first schema-valid message (issue #19816). Delete/ignore any pre-existing `-o` file before each run — codex does not truncate it on failure, so a stale success artifact from a prior attempt can masquerade as fresh output.
4. **Local validate-and-resume loop is still worth having, but expect it to fire rarely**: with strict server-side constrained decoding, a completed turn's final message should always parse and conform; residual risks are token-limit truncation, provider quirks (`--oss` local servers may not enforce `text.format`), and codex versions/paths where the schema is dropped. On local validation failure after exit 0: `codex exec resume <THREAD_ID> --output-schema schema.json --json "<repair prompt: reply ONLY with JSON matching the schema>"` (thread id from the `{"type":"thread.started","thread_id":...}` event; `--output-schema` is global so it applies to `resume`; the resumed turn re-attaches the schema — lib.rs loads it on both the fresh and resume paths, lines ~747/767). Cap at 1-2 repair attempts. Note `resume` requires session persistence — do not combine with `--ephemeral`.
5. **Transient failures** (turn.failed whose message is not `invalid_json_schema`/usage-limit): codex already burned 5 stream retries internally with backoff, so add at most 1 agent-level re-run (fresh `codex exec` or `resume --last`) with meaningful backoff; map repeated failure to null. For `usage_limit_reached`/`quota` messages, back off long or fail fast — internal retries don't apply and immediate re-runs will fail identically.
6. **pipeline() mapping**: exit 0 + parseable/validating last message → value; anything else → null, with reason taken from the last `turn.failed.error.message` (fallback: last `error` event; fallback: stderr tail for pre-flight exits, which print human text to stderr and never emit JSONL).

## KEY FACTS
- codex exec --output-schema is enforced purely server-side: the schema is sent verbatim on every Responses API request as text.format = {name: "codex_output_schema", type: "json_schema", strict: true, schema: <user schema>}; asserted exactly by integration test exec_includes_output_schema_in_request [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/tests/suite/output_schema.rs]
- Client-side, the schema file is only checked to be readable and valid JSON (serde_json parse) in load_output_schema(); no JSON-Schema validation of the file or of the model output exists anywhere (no jsonschema validator crate in the workspace; schemars is only for config schema generation) [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/lib.rs]
- A schema violating the structured-outputs strict subset (e.g. root type array) produces HTTP 400 invalid_request_error/invalid_json_schema on param text.format.schema; this maps to CodexErr::InvalidRequest which is_retryable()==false, so the turn fails immediately with zero internal retries [https://github.com/openai/codex/issues/16552 and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/codex-api/src/api_bridge.rs (lines 63-82), https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/protocol/src/error.rs (is_retryable, lines 173-210)]
- Retryable errors (stream drops, 5xx, timeouts, UnexpectedStatus) are internally retried up to stream_max_retries (default 5, configurable per provider, cap 100) with exponential backoff plus one WebSocket-to-HTTPS fallback; retry notices surface as JSONL {"type":"error","message":"Reconnecting... n/5"} events that do NOT indicate failure [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/core/src/responses_retry.rs and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/model-provider-info/src/lib.rs (DEFAULT_STREAM_MAX_RETRIES=5)]
- Failure surfaces in --json mode as {"type":"error","message":...} followed by {"type":"turn.failed","error":{"message":...}}, and the process exits 1 (error_seen is set on ServerNotification::Error with will_retry=false or TurnCompleted with status Failed/Interrupted, then std::process::exit(1) after print_final_output) [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/lib.rs (run loop ~lines 985-1050) and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/event_processor_with_jsonl_output.rs (TurnStatus::Failed arm)]
- Exit codes: 0 = turn completed; 1 = every failure class (turn.failed, interrupt, auth/config/MCP/git-check/schema-file preflight errors, anyhow errors); 2 = clap usage error; no finer-grained codes exist; pinned by tests asserting .code(1) for server response.failed and required-MCP init failure [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/tests/suite/server_error_exit.rs and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/tests/suite/mcp_required_exit.rs]
- On turn.failed the -o/--output-last-message file is not written and not truncated (stale file from a prior run survives); on success it contains exactly the last agent_message text (a raw JSON string when --output-schema is used) [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/event_processor_with_jsonl_output.rs (print_final_output, emit_final_message_on_shutdown) and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/event_processor.rs (handle_last_message)]
- The schema is attached to every sampling request in the turn (TurnContext.final_output_json_schema cloned into each Prompt), so intermediate progress agent_message items are also schema-shaped JSON; consumers must take the LAST agent_message, not the first schema-valid one (open bug #19816) [https://github.com/openai/codex/issues/19816 and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/core/src/session/turn.rs (build_prompt, lines 1104-1113)]
- --output-schema is a global clap arg and is loaded on both the fresh-turn and resume paths, so `codex exec resume <THREAD_ID> --output-schema schema.json <prompt>` re-attaches the schema for a repair turn; thread id comes from the {"type":"thread.started","thread_id":...} JSONL event; resume is incompatible with --ephemeral (no persisted session) [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/cli.rs (global=true, line 53) and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/lib.rs (load_output_schema called at ~747 and ~767)]
- output_schema_strict is hardwired true for exec sessions (only internal guardian-reviewer sessions are non-strict), and WireApi has only the Responses variant in 0.142.x (Chat Completions wire API removed), so there is no alternate response_format path [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/core/src/session/turn.rs (lines 1110-1112) and https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/model-provider-info/src/lib.rs (enum WireApi, lines 53-60)]
- Sandbox denials and auto-rejected approvals do NOT affect the exit code: exec mode rejects all approval/elicitation server requests and the command failure is fed back to the model; the turn still completes with status Completed and exit 0 [https://github.com/openai/codex/blob/rust-v0.142.4/codex-rs/exec/src/lib.rs (handle_server_request, ~lines 1700-1786)]

## UNCERTAINTIES
- Empirical run with an impossible schema (e.g. {"const": false}) was not executed — this session is a read-only planning phase and codex exec writes ~/.codex session state and consumes API quota. The reported JSONL sequence and exit code for the 400 path are derived from source tracing plus issue #16552's captured error payload and the repo's own .code(1) integration tests, not a live capture. A follow-up implementation-phase task could run: codex exec --json --ephemeral --skip-git-repo-check --output-schema bad.json 'say hi' and diff against the predicted event stream.
- Invalid-credential (HTTP 401) classification: 401 falls into the UnexpectedStatus branch of map_api_error, and CodexErr::UnexpectedStatus is marked retryable in 0.142.4 — so a bad API key may burn up to stream_max_retries (5) backoff attempts before turn.failed/exit 1, unless an auth-refresh path (RefreshTokenFailed, non-retryable) intervenes first. Not traced end-to-end; either way the final exit code is 1.
- Whether --oss local providers (LM Studio/Ollama via Responses-shaped requests) actually enforce text.format constrained decoding is provider-dependent; codex sends the field regardless. For non-OpenAI backends, local validation in the orchestrator is the only real guarantee.
- Token-limit truncation of a structured response (response.incomplete) was not traced to a specific event/exit outcome; presumed to surface as a completed-or-failed turn depending on where the stream ends. Low probability but a reason to keep agent-level JSON validation.
- Version drift: findings pinned to rust-v0.142.4 (matches installed binary; 0.142.5 is latest, released 2026-07-01). The exec crate was recently rearchitected onto the in-process app-server client, so line numbers and event plumbing may shift quickly in future releases; the JSONL event schema (thread.started/turn.started/turn.completed/turn.failed/item.*/error) and exit-code contract have been stable across this rearchitecture.
- GitHub code search via unauthenticated API returned empty results; searches were re-run with authenticated gh CLI. Issue search for 'output-schema' returns many unrelated hits; the three cited issues (#16552, #19816, #15451) are the load-bearing ones found — there may be additional closed duplicates (#2459, #4181, #4776 referenced in #16552) not individually reviewed.
