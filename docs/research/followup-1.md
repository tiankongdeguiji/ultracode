# GAP: Structured-output enforcement and exact JSON/stream-json event schema for qodercli headless mode

# Qoder CLI (qodercli) headless structured-output & JSON envelope research — 2026-07-04

## Headline findings

1. **Native JSON-Schema enforcement EXISTS in qodercli but is completely undocumented**: the CLI accepts a **`--json-schema '<serialized JSON Schema object>'`** flag. Hard evidence: the official Python SDK (`qoder-agent-sdk` 1.0.5 on PyPI, wheel `qoder_agent_sdk/_internal/transport/subprocess_cli.py`) builds its command as `[qodercli, --print, --output-format, stream-json, --input-format, stream-json, ...]` and appends `cmd.extend(["--json-schema", json.dumps(schema)])` when `options.output_format == {"type": "json_schema", "schema": {...}}` (source: https://pypi.org/pypi/qoder-agent-sdk/json → wheel https://files.pythonhosted.org/packages/28/19/709c1c99a55df06915715bb30776eeada14319d434bf9bbf5ea20754edcd/qoder_agent_sdk-1.0.5-py3-none-manylinux_2_17_x86_64.whl). The schema-conforming value comes back as a **`structured_output` field on the terminal `result` message**, and exhausted validation retries produce result **`subtype: 'error_max_structured_output_retries'`**. No docs page, no `docs.qoder.com/llms-full.txt` line, and no settings.json key mentions this flag — it is SDK-source-only knowledge. (qodercli is a Claude Code-lineage fork — the Python SDK contains `RateLimitType = "seven_day_opus" | "seven_day_sonnet"` and links to `docs.qoder.com/en/docs/qoder-code/rate-limits`; Claude Code's own `claude -p --output-format json --json-schema "$SCHEMA"` pattern is documented third-party at https://stevekinney.com/courses/self-testing-ai-agents/structured-cli-output-as-pipeline-glue, so identical print-mode semantics are highly likely.)

2. **Per-SDK exposure is asymmetric**:
   - **Python SDK (`pip install qoder-agent-sdk`, v1.0.5, Alpha)**: PUBLIC option. `QoderAgentOptions.output_format: dict | None` — docstring: `# Output format for structured outputs (matches Messages API structure)\n# Example: {"type": "json_schema", "schema": {"type": "object", "properties": {...}}}`. Result surfaces as `ResultMessage.structured_output: Any`. NOT documented in https://docs.qoder.com/en/cli/sdk/python/references.md (grep for `output_format`/`json_schema` = 0 hits).
   - **TypeScript SDK (`@qoder-ai/qoder-agent-sdk` 1.0.11 npm, modified 2026-06-29, paired `qoderCliVersion: 1.0.33`)**: NOT in the public `Options` type (`dist/types/options.d.ts` has no outputFormat/jsonSchema key). The internal plumbing exists — `QueryRunner(transport, isSingleTurn, canUseTool, hooks, abortController, sdkMcpServers, jsonSchema, initConfig, ...)` sends `jsonSchema` in the **`initialize` control_request** (`SDKControlInitializeRequest.jsonSchema?: Record<string, unknown>` in `dist/protocol/control.d.ts`) — but the local Node `query()` call site hardcodes `void 0` for that parameter (verified in minified `dist/index.js`: `new ae(se,ne,t.canUseTool,t.hooks,t.abortController,p,void 0,es,...)`). Only `BrowserQueryOptions.jsonSchema` (WebSocket transport, `dist/types/browser.d.ts`) exposes it publicly. **Workaround for TS**: `extraArgs: { 'json-schema': JSON.stringify(schema) }` — buildArgs appends `--${key} <value>` for every extraArgs entry (`for(let[r,i]of Object.entries(t.extraArgs))e.push(\`--${r}\`),i!==null&&e.push(i)`), and `SDKResultSuccess.structured_output?: unknown` is already in the protocol types (`dist/protocol/messages.d.ts`), so the value parses through the normal message stream. Types `OutputFormatType = 'json_schema'`, `JsonSchemaOutputFormat = { type: 'json_schema'; schema: Record<string, unknown> }`, `OutputFormat = JsonSchemaOutputFormat` are exported from `dist/protocol/common.d.ts` (napm tarball: https://registry.npmjs.org/@qoder-ai/qoder-agent-sdk/-/qoder-agent-sdk-1.0.11.tgz).

3. **No public CLI docs flag**: https://docs.qoder.com/en/cli/using-cli.md documents print mode only as: `-p` (run non-interactively; example `qodercli -q -p hi`), `--output-format` = `text | json | stream-json`, plus `-w`, `-c`, `-r`, `--allowed-tools`, `--disallowed-tools`, `--max-turns`, `--yolo`. https://docs.qoder.com/qoderwake/cli-reference.md adds subcommands (`login/status/commit/update/config/mcp/agents/skills/plugins/hooks/wiki/rollback/feedback`), `-m <model>`, `--attachment`, `--remote`, `--mcp-config`, `--agent`, and says "For the complete list, run `qodercli --help`".

## Exact stream-json envelope (`--print --output-format stream-json`)

JSONL, one JSON object per line; `dist/types/messages.d.ts` states verbatim: "SDKMessage — JSONL line format the CLI emits to stdout." Message union (protocol wire, from `dist/protocol/messages.d.ts` + docs https://docs.qoder.com/en/cli/sdk/references.md):

- **init**: `{type:'system', subtype:'init', uuid, session_id, qodercli_version, protocol_version?, apiKeySource:'none|user|project|org|temporary|oauth', cwd, model, permissionMode, tools:[...], slash_commands:[...], output_style, agents?, skills:[...], plugins:[{name,path,source?}], mcp_servers:[{name,status}], fast_mode_state?}`. `protocol_version` handshake: `WIRE_PROTOCOL_VERSION = "1.0.0"`; SDK refuses cross-major.
- **assistant**: `{type:'assistant', uuid, session_id, parent_tool_use_id, message:{role:'assistant', content:[{type:'text'|'tool_use'|'thinking',...}]}}`; may carry `error: 'authentication_failed'|'billing_error'|'rate_limit'|'invalid_request'|'server_error'|'unknown'|'max_output_tokens'` (`SDKAssistantMessageError`).
- **user** (tool results echoed back): content items include `{type:'tool_result', tool_use_id, content, is_error?}`.
- **result (success)** — the terminal envelope, exact field names: `{type:'result', subtype:'success', duration_ms:number, duration_api_ms:number, is_error:boolean, num_turns:number, result:string, stop_reason:string|null, total_cost_usd:number, usage:NonNullableUsage, modelUsage:Record<string,ModelUsage>, permission_denials:SDKPermissionDenial[], error_code?:number, structured_output?:unknown, deferred_tool_use?:unknown, terminal_reason?:string|null, fast_mode_state?, uuid, session_id}`. NOTE: the session id key is exactly **`session_id`** (snake_case) on every message; **`modelUsage`** is camelCase on the wire (Python parser reads `data.get("modelUsage")` into `model_usage`); everything else snake_case.
- **result (error)**: same common fields plus `errors: string[]`, `subtype: 'error_during_execution' | 'error_max_turns' | 'error_max_budget_usd' | 'error_max_structured_output_retries'`. (The docs page https://docs.qoder.com/en/cli/sdk/references.md omits `error_max_budget_usd`; npm typings include it.)
- `usage` (`NonNullableUsage`): `{input_tokens, output_tokens, cache_creation:{ephemeral_1h_input_tokens, ephemeral_5m_input_tokens}, cache_creation_input_tokens, cache_read_input_tokens, inference_geo, server_tool_use:{web_fetch_requests, web_search_requests}, service_tier, speed, iterations:[], request_id?, context-ratio field (0~1)}`. `ModelUsage` per model: `{inputTokens, outputTokens, cacheReadInputTokens, cacheCreationInputTokens, webSearchRequests, costUSD, contextWindow, maxOutputTokens}` (camelCase).
- **stream_event** (only with `--include-partial-messages`): `{type:'stream_event', uuid, session_id, parent_tool_use_id, event:{type:'message_start|content_block_start|content_block_delta|content_block_stop|message_delta|message_stop', delta?:{type:'text_delta'|'input_json_delta'|'thinking_delta', text?/partial_json?/thinking?}}}` (https://docs.qoder.com/en/cli/sdk/streaming-output.md).
- Other system subtypes: `compact_boundary`, `status` (`status:'compacting'|null`), `mcp_status_change`, `api_retry` (`attempt, max_retries, retry_delay_ms`), `permission_denied`; plus `tool_progress` messages `{tool_use_id, tool_name, parent_tool_use_id, elapsed_time_seconds, task_id?, uuid, session_id}`, task/hook lifecycle messages, `keepalive`/`keep_alive` and `streamlined_text` (filtered by SDKs).
- **Control protocol** (bidirectional, requires `--input-format stream-json`): `{type:'control_request', request_id, request:{...}}` / `{type:'control_response', response:{subtype:'success'|'error', request_id, response?|error, code?, pending_permission_requests?}}` / `{type:'control_cancel_request'|'control_cancel', request_id}`. The `initialize` request accepts `{type:'initialize', model?, cwd?, allowedTools?, disallowedTools?, mcpServers?, agents?, plugins?, systemPrompt?, appendSystemPrompt?, excludeDynamicSections?, permissionMode?, maxThinkingTokens?, enableFileCheckpointing?, sdkMcpServers?, jsonSchema?, promptSuggestions?, supportsCatalogReadyInitialize?, initializeTimeoutMs?, agentProgressSummaries?, hooks?}` — so a raw-process orchestrator can also inject the schema via initialize instead of the flag. Permission prompting arrives as inbound `control_request` `{subtype:'can_use_tool', tool_name, input, tool_use_id, permission_suggestions?, ...}` when spawned with `--permission-prompt-tool stdio`.

## `--output-format json` (non-stream) envelope

**Not documented anywhere** (docs only name the mode; https://forum.qoder.com/t/9599 = single-post feature request "Qoder CLI 后续版本支持 -p --output-format stream-json" by 晓文_陈, 2026-06-08, zero replies, no envelope details — checked via https://forum.qoder.com/t/9599.json). Given the shared Claude Code lineage and that stream-json's terminal line IS the result object, the json mode almost certainly prints the single `type:'result'` object described above (with `structured_output` when `--json-schema` was passed) — but this remains unverified because no qodercli binary was installable in this read-only phase.

## Exit codes & failure semantics

- Python SDK: after transport close, `returncode != 0` ⇒ `ProcessError("Command failed with exit code {n}", exit_code=n)`; special-cased **`CLI_EXIT_CODE_AUTH_ERROR = 41`** (fires auth-expired callback instead of raising). So: **0 = clean exit; non-zero = process failure; 41 = authentication failure**.
- Semantic (in-band) failure is signaled by the result message: `is_error: true` + error `subtype` + `errors: string[]` (and `permission_denials` lists blocked tool calls). TS SDK `onAuthExpired` also triggers on assistant `error:'authentication_failed'` or non-zero exit + auth-ish stderr. Whether an `is_error` result also forces a non-zero process exit is unverified.
- `ExitReason` (session-end): `'clear'|'resume'|'logout'|'prompt_input_exit'|'other'|'bypass_permissions_disabled'`.

## Tool-injection / emulation paths (both SDKs)

- **In-process MCP tools**: TS `tool(name, description, zodRawShape, handler)` + `createSdkMcpServer`; Python `tool` + `create_sdk_mcp_server` (both exported; Python auto-converts TypedDict/py types → JSON schema via `_python_type_to_json_schema`). Registered via `mcpServers`, invoked over the control/MCP channel — a hand-rolled `StructuredOutput` tool is fully implementable (docs: https://docs.qoder.com/en/cli/sdk/references.md#tool, https://docs.qoder.com/en/cli/sdk/mcp.md).
- **canUseTool** (`CanUseTool` callback; `--permission-prompt-tool stdio` under the hood) exists in both SDKs → can intercept/deny/mutate any tool call (https://docs.qoder.com/en/cli/sdk/permissions.md).
- **Hooks**: full event set incl. `Stop`, `StopFailure`, `SubagentStop`, `PostToolUse`, `UserPromptSubmit`; hook JSON output `decision:"block"` + `reason` forces the agent to continue — a Stop-hook validator is a viable retry loop. The CLI even has an internal **`StructuredOutput` tool** used by `agent`-type hooks: "The sub-agent must call the `StructuredOutput` tool with `{ ok: boolean, reason?: string }`" (https://docs.qoder.com/en/cli/hooks.md, llms-full.txt line 1526) — proving first-class structured-output tooling exists inside the binary.

## Recommended per-host structured-output strategy

| Host / entry point | Strategy | Mechanism | Result extraction | Failure signal |
|---|---|---|---|---|
| Python SDK `qoder-agent-sdk>=1.0.5` (**primary recommendation**) | **Native** | `QoderAgentOptions(output_format={"type":"json_schema","schema":SCHEMA})` | `ResultMessage.structured_output` | `ResultMessage.subtype=='error_max_structured_output_retries'`, `is_error`, `errors[]`; `ProcessError.exit_code` (41=auth) |
| Raw CLI headless `qodercli --print` | **Native (undocumented flag)** | `--json-schema "$(cat schema.json)" --output-format stream-json --input-format stream-json` (stream-json is the SDK-proven path); parse last `type=="result"` JSONL line | `result_line.structured_output` | same subtypes + process exit code |
| TS SDK `@qoder-ai/qoder-agent-sdk@1.0.11` | **Native via escape hatch** | `options.extraArgs = {'json-schema': JSON.stringify(SCHEMA)}` (public Options lacks the key; browser/WebSocket path has `BrowserQueryOptions.jsonSchema`) | `SDKResultSuccess.structured_output` (typed) | `SDKResultError.subtype`, `is_error` |
| Raw stream-json control protocol (no SDK) | Native, alt route | send `{type:'control_request',request_id,request:{type:'initialize', jsonSchema: SCHEMA, ...}}` before first user message | same result line | same |
| Any host, belt-and-suspenders | **Injected tool + validate-retry fallback** | in-process MCP `StructuredOutput` tool (zod/TypedDict) + `allowedTools` + prompt mandate; or Stop-hook validator with `decision:"block"`; orchestrator-side JSON-Schema validation of `structured_output`/`result` with N resume retries (`--resume <session_id>`) | tool call args / re-parsed `result` | validator |

Recommendation for the orchestrator's `agent({schema})`: use the native path everywhere (Python SDK option, TS extraArgs, or raw `--json-schema`), ALWAYS validate `structured_output` orchestrator-side anyway (flag is undocumented ⇒ treat as unstable), map `error_max_structured_output_retries` to a small resume-based retry budget (1–2), and treat exit 41 as a terminal auth failure, other non-zero exits as infra retries.

## Misc facts for the implementation

- SDK-spawned sessions always run `--print --output-format stream-json --input-format stream-json`, plus forced `--setting-sources ""` (isolation) and `--disable-builtin-skills` in the Python SDK; TS adds `--disable-builtin-skills` unconditionally too.
- Model tiers for `--model`: `auto|ultimate|performance|efficient|lite` (~1.0x/…/0.3x credit multipliers, https://docs.qoder.com/en/cli/sdk/references.md).
- CLI binary distribution: Python wheels **bundle** the binary at `qoder_agent_sdk/_bundled/qodercli` (131 MB, platform wheels for macOS arm64/x86_64, manylinux/musllinux aarch64/x86_64, win_amd64) — installing the wheel is the easiest headless way to obtain a known-good qodercli. npm postinstall downloads from `https://download.qoder.com/qodercli/releases/<version>` (CN mirror `https://static.qoder.com.cn/qoder-cli-cn/releases`), version pinned by `package.json.qoderCliVersion` (1.0.33 for SDK 1.0.11); env `QODER_SKIP_DOWNLOAD=1`, `QODER_CLI_MIRROR`, `QODER_CLI_SITE=cn`.
- Auth for headless: PAT via `QODER_PERSONAL_ACCESS_TOKEN` (generate at qoder.com/account/integrations) or reuse of local `qodercli login` state; SDKs write PAT to a one-time auth payload file, not argv (https://docs.qoder.com/en/cli/sdk/python/quick-start.md).
- No settings.json key for structured output exists (docs sweep of llms-full.txt for `schema|structured|json_schema|output-schema` found only hooks/MCP/elicitation matches).
- TS typings carry `// Copyright 2026 Google LLC` headers on protocol files while Python types carry Claude-lineage strings — mixed Gemini-CLI/Claude-Code heritage; the `--json-schema` flag name matches Claude Code's print-mode flag exactly.

## UNCERTAINTIES
- The exact top-level envelope of `qodercli -p --output-format json` (non-stream) is documented nowhere and could not be executed (no binary installable in this read-only phase). Inferred to be the single type:'result' object (identical to the terminal stream-json line, Claude Code lineage), but unverified — verify with one live run before hard-coding the parser.
- --json-schema is only proven in combination with --print --output-format stream-json --input-format stream-json (the SDK spawn path). Its behavior with --output-format json or plain text mode is untested; also unverified whether the flag exists in older CLI builds (pairing evidence: SDK 1.0.11 ↔ qodercli 1.0.33, Python SDK 1.0.5).
- The internal retry count behind subtype 'error_max_structured_output_retries' is CLI-internal and not configurable/documented anywhere found; no settings key or flag controls it.
- Whether the CLI process exits non-zero when the result message has is_error:true (vs exit 0 with in-band error subtype) is unverified; only exit 41 (auth) and generic non-zero=ProcessError semantics are confirmed from SDK source.
- Docs pages conflict with npm typings on the error subtype list: https://docs.qoder.com/en/cli/sdk/references.md omits 'error_max_budget_usd' (and Python docs show only a truncated subtype list); the npm protocol typings are the more complete source.
- docs example uses `qodercli -q -p hi` — the -q flag is undocumented (presumably quiet); meaning unconfirmed.
- TS SDK public Options lacking output_format may change soon (Python SDK 1.0.5 already has it; TS 1.0.11 published 2026-06-29 does not) — re-check @qoder-ai/qoder-agent-sdk on npm before finalizing the TS integration.
- Whether the main-session model can call the internal StructuredOutput tool directly (outside agent-type hooks) is unknown; it is not listed in the built-in tool list of the SDK references.
