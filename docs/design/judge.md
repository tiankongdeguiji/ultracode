# Portable Ultracode — Recommended Architecture

## 0. Delivery form (plugin vs command vs skill)

**All three, in layers — and the layers are different artifacts, not alternatives.** The *doctrine* (when/how to orchestrate, the dialect, quality patterns) is a single canonical **Agent Skill** (`skill/ultracode/SKILL.md` + `references/`) copied per host into `.agents/skills/` / `.qoder/skills/` / `.claude/skills/` — skills are the only cross-host surface with implicit model-triggered invocation and progressive disclosure. The *engine* (runtime, fan-out, journal/resume) is a single **npm package `ultracode`** shipping a CLI and an MCP stdio server (`ultracode mcp`) — a skill cannot host a process, and a single blocking MCP tool dies at every host's 300–600s timeout. *Delivery* is a **plugin where plugins exist** (Codex plugin bundling skill + MCP registration; Qoder plugin bundling skill + `uc-*` workflows + agent defs) and an **installer command everywhere else** (`ultracode install <host>`, merge-not-overwrite, idempotent). The *command* surface is **emergent, not authored**: skills already register as `$ultracode` (Codex) and `/ultracode` (Qoder/Cursor/others), so no separate command artifact exists. Codex custom prompts are deprecated; a bare command can carry neither doctrine nor a runtime — so "command-only" and "MCP-only" are both wrong answers.

## 1. Repository & package (ruling: monorepo + generated mirrors)

One git repo → one npm package `ultracode` (TypeScript, ESM, Node ≥ 20, deps only acorn/ajv/commander/@modelcontextprotocol-sdk, zero native deps). It contains: `src/engine`, `src/backends`, `src/store`, `src/exec`, `src/budget`, `src/cli`, `src/mcp`, `src/installer`, the canonical `skill/`, canonical `workflows/` templates (`uc-*`), `dist-codex/` and `dist-qoder/` plugin build outputs, and mock-first tests with golden NDJSON fixtures. The two marketplace repos (`ultracode-marketplace-codex`, `ultracode-marketplace-qoder`) are **CI-generated read-only mirrors** stamped with `GENERATED-FROM: <sha>` — never hand-edited. Ruling rationale: skill/template drift across hosts is the #1 failure mode; one source of truth plus mechanical mirrors eliminates it, and the npm binary is mandatory everywhere anyway because every MCP registration points at `ultracode mcp`.

## 2. Execution model (ruling: detached runner from day one; this dissolves the MCP-in-v1 dispute)

**Every run executes in its own detached `ultracode-runner` process; there is no daemon; the on-disk run store is the only coordination plane.** CLI and MCP server are both stateless launchers/readers over `.ultracode/runs/<wf_...>/{manifest.json, script.js, args.json, journal.jsonl, events.jsonl, output.json, agents/**}` (single-writer runner, atomic manifest swaps, O_APPEND event/journal lines, 5s heartbeat, orphan reaping).

This resolves the sharpest conflict. The minimalist cut the MCP triad as "the single largest complexity item"; the core-engine and host designs kept it. Ruling: **the MCP triad IS in v1** — the user runs in ultracode mode and fire-and-forget across host turns is core value, not polish — but the minimalist is right about *sequencing*: the CLI ships and is proven live first, and Design 1's architecture makes the triad cheap, because run persistence, detachment, and orphan handling are built for the CLI anyway (resume needs them). The MCP server then adds only ~4 thin read/launch handlers.

CLI ergonomics reconcile Design 1 (detach default) and Design 3 (foreground kill-switch): `ultracode run` **attaches in the foreground by default** (tails events, exit code mirrors run status); execution is nonetheless always in the detached runner, so **Ctrl-C = explicit stop (SIGTERM→5s→SIGKILL of all agent process groups, journal preserved, partial output.json)** while **SIGHUP/connection loss = run survives**, re-attachable via `ultracode status --watch`. `--detach` prints runId and returns.

## 3. Engine core (adopt Design 1 wholesale, with Design 3's safety defaults)

- **Sandbox (ruling — unanimous, research-confirmed): hardened `node:vm`** inside the runner process: frozen intrinsics, exact Qoder ban messages for `Date.now`/no-arg `Date`/`Math.random`, deleted `WebAssembly/ShadowRealm/Atomics/SharedArrayBuffer`, fixed global set (`agent, parallel, pipeline, phase, log, workflow, args, budget, console, setTimeout/clearTimeout`), JSON round-trip at the host boundary, 30s sync timeout, 512KB script cap. quickjs-emscripten is rejected on a hard technical ground: Asyncify allows one suspension at a time, and the dialect requires arbitrary concurrent guest awaits over `agent()` (`parallel`, bare `Promise.all`). SES/isolated-vm rejected (compat risk / native dep) for zero payoff under the trust model: model-authored + user-reviewed scripts whose only capability is `agent()`. Review is *enforced*, not assumed: `run` prints meta + the static agent inventory and requires confirmation unless `--yes`; `--dry-run` executes on the mock backend for a free rehearsal.
- **Dialect**: acorn-parsed pure-literal `export const meta`; `agent/parallel/pipeline/phase/log/workflow/budget` with the exact Qoder semantics (allSettled barrier → null + failures[]; pipeline no inter-stage barrier, `stage(prev, item, i)`, throw/null drops item; per-call `phase` option; retries clamped 0–5; `stallMs` watchdog; `skip`). `workflow()` = one nesting level, shared semaphore/counter/signal/budget.
- **Caps (ruling reconciling 1000 vs 50)**: hard lifetime ceiling 1000 agents (spec-fidelity error string) but **soft default `--max-agents 50`**, semaphore `min(16, max(2, cpus-2))`, 4096 items/call, per-agent timeout 20m, wall-clock 60m — every cap trip lands in failures[] and the first line of the report (**no silent caps**).
- **Journal (ruling: `u1:` keys, not byte-compatible `v2:`)**: hash-chain `key_n = "u1:" + sha256(prev \0 prompt \0 stableStringify({agentType, isolation, model, effort, schema, backend, cwd}))`. Backend/effort MUST be in the key: omitting them (byte-compat with Qoder) would silently replay wrong cached results after a backend switch on resume. Journals never interchange with Qoder's on-disk runs, so byte-compat buys nothing. Write path from the first run-store milestone; **prefix-replay resume ships in v1** (first-miss-disables-rest, cross-process by construction) — the minimalist's cut is rejected because long ultracode-mode runs make full re-spend a real cost, and the read path is small once keys are locked by determinism tests.
- **Budget**: real `{total, spent(), remaining()}` from normalized usage events; **dispatch-gate** enforcement (in-flight agents finish, next `agent()` throws); `+500k`/`2m` parsing; chars/4 estimate flagged `estimated:true` where a backend omits usage; live cost line per completion.

## 4. Backends (ruling: 5 in v1 — mock, codex, qoder, claude, gemini; long tail deferred)

`BackendAdapter` interface per Design 1 (probe / checkSchema / buildSpawn / buildResume / stateful NDJSON parser / classifyExit / extractUsage). v1 ships: **mock** (fault-injecting test substrate, first-class), **codex** (prompt via stdin; **never `-o`** — take the LAST `agent_message`, issue #19816, eliminating the stale-file trap the minimalist worked around; `turn.failed` authoritative, `Reconnecting…` notices benign; strict-subset schema pre-validation with fail-fast on deterministic 400s), **qoder** (worker adapter: `--print --output-format stream-json --json-schema`, exit 41 = auth, PAT-stateless; feature-detect fallback to emulated loop since the flag is undocumented), **claude** (near-free: qoder envelope's ancestor), **gemini** (the canonical emulated-schema backend, proving the prompt-contract + ajv + fresh-spawn repair loop). **cursor/copilot/opencode/amp are deferred** — a genuine risk-based cut, not effort-saving: their envelopes were pinned least firmly, each adapter joins a maintenance treadmill against weekly/daily CLI releases, and two native-schema + one emulated backend already exercise every code path. The interface is designed for them; they land demand-driven post-v1.

**Structured output — the single most important rule (unanimous)**: native flags are accelerators, never the source of truth. Engine-side ajv (2020-12) validates against the ORIGINAL schema on every path; ≤2 resume-based repair turns (fresh-spawn where no resume); repair budget independent of task retries; untransformable Codex-strict schemas rejected at parse time with the offending path, before any tokens burn. Flag drift therefore degrades quality (more retries), never correctness.

**Auth policy (unanimous, doctor-enforced at every run start)**: Codex = shared `CODEX_HOME` + per-process `CODEX_API_KEY`; ChatGPT-OAuth credentials → refuse fan-out (concurrency 1; `--force-oauth-fanout` caps at 3 with a pre-warm serial agent); never copy auth.json. Qoder = `QODER_PERSONAL_ACCESS_TOKEN`, with a doctor probe for the `/login`-beats-env-PAT trap. Also v1: the **silent no-op detector** — codex exec auto-rejects approvals yet exits 0, so the adapter surfaces declined command/file actions as `failure:{kind:'cap', msg:'N actions auto-rejected — wrong sandbox mode?'}`, and doctrine requires mutating agents to return a `changed_files` schema. Default subagent sandbox is **read-only**; `workspace-write` is explicit per run; the engine never passes `--yolo`/`danger-full-access`.

## 5. MCP surface (v1, milestone-gated)

Exactly `workflow_start` (<1s return) / `workflow_status` (long-poll, `waitSeconds` default 25 clamped 50, event-offset cursor, `notifications/progress` during waits) / `workflow_result` / `workflow_stop` / `workflow_list`. **Never declare `taskSupport`** (`required` throws client-side in Qoder; nothing polls Tasks). 50s sits under every host default (Codex 300s — 60s only on ≤v0.98 legacy — Qoder/Gemini 600s). Because runs are detached, Codex's timeout-orphaning behavior costs nothing: the server is disposable, the run store is truth. Server `instructions` ≤512 chars telling the model: start returns fast, keep polling, a timed-out poll is harmless — re-poll with the same runId.

## 6. Qoder (ruling: ride native; external engine is fallback + worker, not primary)

Qoder 1.0.37 ships a faithful native engine (same dialect, hash-chain resume, fire-and-forget with task-notification completion, per-agent skip/retry UI) — replacing it would duplicate a deeper integration than we can build. The plugin ships `uc-*` workflow templates (shared dialect, `budget.total ?? args?.budgetTokens` shim for the stubbed budget; `agentType:'uc-xhigh'` agent defs for per-call effort routing; never `deep-research`, nothing named `Workflow`), the skill, and an `always_on` rule (~150 tokens) providing the keyword→orchestrate wire Qoder lacks. `ultracode doctor` probes the remote feature gate ("Workflow feature gate is disabled."), caches the result with TTL, and flips dispatch to the already-registered MCP triad when the gate is off. What Qoder actually lacks is **doctrine** — the minimalist's strongest point, adopted fully: the engine is inert without the skill telling the model when and how to orchestrate.

## 7. Ultracode mode (standing opt-in)

Three cooperating carriers: sticky-for-session rule in SKILL.md (keyword "ultracode" / budget token anywhere → route every substantive task through a workflow until "ultracode off"); machine-readable `.ultracode/mode` file toggled by `ultracode mode on|off` or the model; and a marker-guarded **AGENTS.md snippet** (installer-appended, idempotent; Gemini via `contextFileName`) as the durable trigger surviving sessions and compaction, plus the Qoder `always_on` rule. Doctrine includes the anti-trigger rubric (no workflows for single-file edits, <3 independent units, tasks needing mid-run user input) — over-triggering is how users uninstall.

## 8. Portability of workflow scripts

Canonical `.ultracode/workflows/*.workflow.js`; `ultracode sync` makes stamped real copies (not symlinks) into `.claude/workflows/` and `.qoder/workflows/` with hash headers, `sync --check` drift detection, `--adopt` reclaim. `ultracode lint` enforces the portable subset: pure-literal meta, determinism bans, Codex-strict schemas (the safe intersection), budget-via-args shim, `uc-` prefix, no per-call effort. The dialect itself is the portability layer — same script text runs on Claude Code native, Qoder native, and this engine.

## V1 CUTS
- Windows support (process-group kill semantics differ fundamentally; explicit startup error rather than a half-working kill path that orphans billing processes; v2+ with Job Objects/taskkill /T)
- cursor, copilot, opencode, and amp backend adapters (envelopes least-firmly pinned by research, each joins a weekly-release maintenance treadmill; BackendAdapter interface is designed for them; land demand-driven post-v1)
- quickjs-emscripten / SES sandbox tier (only pays off if untrusted third-party workflow distribution becomes a feature; Asyncify additionally breaks the dialect's concurrent-await semantics; revisit if workflow sharing emerges)
- MCP Tasks extension and any execution.taskSupport declaration (unsupported by every target host; taskSupport:'required' hard-breaks Qoder client-side)
- ChatGPT-OAuth parallel fan-out as a supported mode (refused by doctor policy; --force-oauth-fanout escape hatch capped at concurrency 3 with pre-warm; users on ChatGPT plans get degraded concurrency by design, stated in docs)
- Host hooks integrations (Codex hook-trust friction, no cross-host convergence, zero payoff since the engine journals server-side)
- Byte-compatible Qoder v2: journal keys (deliberately incompatible u1: keys that include backend/effort — correctness over compatibility)
- Symlink-based workflow sync (Windows and unverified registry symlink handling; stamped real copies instead)
- Env-var as the ultracode-mode carrier for the model (hosts don't reliably surface env to the model; engine CLI honors ULTRACODE_MODE but doctrine rides skill + mode file + AGENTS.md)
- Mid-run user input / elicitation inside workflows (workflows cannot ask; encoded as a don't-orchestrate rubric rule instead)
- Unobservable fidelity items implemented lazily (teamName option, exact 1000-entry log-cap edge behaviors) — chase only fidelity users can observe: dialect surface, error strings, output.json shape, cap reporting

## DISAGREEMENTS
- **MCP server in v1 vs CLI-first**: Design 1 and Design 2 ship the workflow_start/status/result triad in v1 as the primary non-Qoder surface; Design 3 defers it to v2 as 'the single largest complexity item', running v1 as a blocking foreground CLI. → RULING: MCP triad IS in v1 (milestone 10 of 14), after the CLI core is proven live. The user's ultracode-mode directive makes cross-turn fire-and-forget core value, and Design 1's detached-runner + on-disk-store architecture (adopted) collapses the minimalist's cost estimate: persistence, detachment, and orphan handling are needed for resume and the CLI anyway, leaving the MCP server as ~4 stateless handlers over the run store. The minimalist's sequencing insight is kept; his cut is not. (confidence: high)
- **Execution model: detached-by-default vs blocking foreground**: Design 1 detaches every run and returns a runId; Design 3 blocks in the foreground so Ctrl-C is a clean kill switch. → RULING: Both, structurally: execution ALWAYS happens in a detached runner process; the CLI attaches in the foreground by default (tails events, mirrors exit code). Ctrl-C = explicit stop with full process-group kill (Design 3's safety requirement); SIGHUP/host-death = run survives and is re-attachable (Design 1's durability requirement). --detach returns immediately; MCP always detaches. (confidence: high)
- **Resume / prefix-replay in v1**: Design 1 ships full hash-chain prefix replay; Design 3 writes the journal but defers the read path to v2 ('a failed demo run is cheaply re-run'). → RULING: Resume ships in v1 (M8). Ultracode-mode runs are long and expensive — full re-spend on a 45-minute failed run is a real cost, not a demo inconvenience. Risk is contained the minimalist's way: journal write path + determinism-of-keys tests land at M4, locking the contract before the read path exists, so resume is a pure additive read path. (confidence: high)
- **Backend breadth: 8 adapters vs 2**: Design 1 ships all eight (codex/qoder/claude/gemini/cursor/copilot/opencode/amp); Design 3 ships mock + codex only ('two hosts prove portability; eight prove nothing extra'). → RULING: Five in v1: mock, codex, qoder, claude, gemini. This covers every code path (native schema strict + native schema permissive + emulated schema + fault injection) and every host the user actually orchestrates from, while the remaining four are a genuine risk cut — least-verified envelopes and the steepest maintenance treadmill — not an effort cut. Interface designed for them; demand-driven post-v1. (confidence: medium)
- **Qoder: native Workflow tool vs external engine**: Designs 2 and 3 ride Qoder's native tool (Design 3 with zero engine at all in v1); Design 1 builds a full qoder adapter without addressing the host question. → RULING: Native tool is primary on Qoder (deeper integration than we can build: task notifications, per-agent skip/retry UI, native resume). The external engine still appears on Qoder in two distinct roles: (a) MCP fallback for accounts with the remote feature gate off (doctor probe, TTL-cached), landing M12 not day one; (b) qoder-as-worker-backend so engine runs on other hosts can spawn qoder workers. The gap Qoder actually has is doctrine, which the skill + always_on rule fill. (confidence: high)
- **Repo topology**: Design 2: single npm package + CI-generated marketplace mirror repos; Design 3 implies single repo; multi-repo implicit alternative. → RULING: Monorepo → one npm package `ultracode`; marketplace repos are CI-generated read-only mirrors with provenance stamps. Skill/template drift across hosts is the top failure mode; one editable source of truth prevents it, and marketplace formats' standalone-repo requirement is satisfied mechanically. (confidence: high)
- **Journal cache-key construction**: Design 3 uses Qoder's byte-compatible 'v2:' keys hashing {agentType, isolation, model, schema}; Design 1 uses 'u1:' keys adding backend, effort, cwd. → RULING: 'u1:' with backend/effort/cwd. Byte-compat with Qoder buys nothing (run dirs never interchange) and is actively wrong: a resume after switching backend or effort would silently cache-hit results produced under different conditions. (confidence: high)
- **Codex -o output file**: Design 3 passes --output-schema with -o and deletes the stale file before each run; Design 1 bans -o entirely and extracts the LAST agent_message from the JSONL stream. → RULING: Never use -o. Deleting stale files mitigates the trap; not creating the trap class eliminates it, and the LAST-message rule is needed anyway for issue #19816 (intermediate messages are also schema-shaped). (confidence: high)
- **Agent-cap default: 1000 vs 50**: Design 1 defaults to the spec-fidelity 1000 lifetime cap; Design 3 defaults --max-agents 50 ('1000 agents on a misconfigured first run is a three-digit bill'). → RULING: Both layers: hard ceiling 1000 with the exact spec error string (fidelity), soft default 50 raised via --max-agents (safety), every trip reported loudly per the no-silent-caps doctrine. Safety default wins because it is doctrine-compliant — a reported cap is not a silent cap. (confidence: high)
- **Run store location**: Design 1: project-local .ultracode/ (git-adjacent, per-project resume); Design 2: ~/.ultracode/runs/ (survives anything, user-global). → RULING: Project-local <cwd>/.ultracode/ with $ULTRACODE_HOME / --home override, and workflow_start accepting cwd. Per-project resume and namespace defensiveness (outside .qoder/ and .claude/) matter more than a global registry; the override covers the global-store preference. (confidence: medium)
- **Worktree isolation and nested workflow() in v1**: Design 1 includes both; Design 3 defers both (demo is read-only; 'parallel mutation deserves its own design pass'). → RULING: Both in v1 at M13, late in the sequence. Parallel mutation (fix-until-green, migrations) is central to the full vision, and worktrees are the only safe way to do it; nesting is one level with shared caps, matching the reference. The minimalist's caution is honored by sequencing them after safety rails (M7) and resume (M8) exist. (confidence: medium)

## USER QUESTIONS
- Publishing identity and reach: is this going public (npm package name `ultracode`, a GitHub org for the two marketplace mirror repos, marketplace listings), or staying internal/private for now? M14's plugin distribution and the installer's marketplace hints depend on the answer.
- Which backend accounts/API keys do you actually have for live testing (CODEX_API_KEY vs ChatGPT plan, Qoder PAT, Anthropic, Gemini), and what per-month budget may CI live-smoke tests and golden-fixture re-records burn? This gates whether M12's adapters can be verified live or must ship fixture-only.
- Default mutation posture: should subagents default to read-only sandbox with explicit --sandbox workspace-write per run (safe, but every fix-it workflow needs a flag), or default to workspace-write since your primary use is fix/migrate workflows in repos you own? This is a risk-tolerance call the research cannot make for you.
- On Qoder, do you accept the native-tool tradeoff — better UX (task notifications, per-agent skip/retry) but NO real token-budget enforcement (Qoder's budget global is a stub, so '+500k' becomes advisory via args) — or do you want the external engine as primary on Qoder too, for uniform budget kill-switch behavior at the cost of the native integration?
